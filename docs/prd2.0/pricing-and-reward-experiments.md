# PPA 2.0 定价与奖励实验方案

## 1. 文档目的

`PPA 2.0` 当前最危险的不是功能不够，而是：

- 定价拍脑袋
- 奖励机制一上来就失控
- 不知道哪种收费单元更成立

这份文档的目标是把价格和奖励都变成可验证实验，而不是一次性做死的规则。

## 2. 实验总原则

### 原则 1：先验证支付意愿，再追求规模

`2.0` 的目标不是一开始赚很多钱，而是确认“有人愿意为结果付钱”。

### 原则 2：奖励要服务供给，不要吞掉收入

奖励机制必须被当作用户获取和留存成本的一部分，而不是福利系统。

### 原则 3：优先做低成本实验

在预算有限的前提下，先做人工 + 半自动验证，不追求完全自动化。

### 原则 4：一次只验证一个核心问题

不要一个实验里同时改价格、改包装、改奖励、改流量来源，否则无法解释结果。

## 3. 要验证的核心问题

### 3.1 定价问题

- 单项目报告是否有人愿意付费
- 连续项目筛选服务是否有人愿意持续付费
- 不同价格带是否显著影响转化

### 3.2 奖励问题

- 外部 AI token 奖励是否足以驱动专业用户参与
- 奖励是否会引来低质量参与
- 奖励池在低预算下能否被控制

## 4. 实验设计框架

建议把实验分成两条主线：

1. **价格实验**
2. **奖励实验**

这两条主线可以并行，但分析时必须分开看。

## 5. 价格实验

## 5.1 实验 A：单项目报告支付意愿测试

### 目标

验证用户是否愿意为“单个公开项目的群体评估结果”付费。

### 实验对象

- 中小软件公司老板
- 投标负责人
- 售前负责人

### 实验方式

准备 3 个价位版本：

- 版本 A：`99 元`
- 版本 B：`199 元`
- 版本 C：`299 元`

### 实验内容

每个版本都销售同一种产品：

- 单项目评估结果
- 可投性建议
- 风险说明
- 成本/工作量区间

### 观察指标

- 咨询率
- 支付转化率
- 支付后反馈
- 是否愿意再次购买

### 判断标准

- 如果 `99 元` 有明显兴趣但 `199+` 完全无人付费，说明价值感还不足
- 如果 `199 - 299 元` 仍有真实支付，说明单项目报告具备基础商品价值

## 5.2 实验 B：连续项目筛选服务测试

### 目标

验证用户是否愿意为持续筛选和跟踪付费，而不是只为一次性报告付费。

### 实验方式

先不做重订阅系统，而是人工或半自动交付 2-4 周服务。

建议测试价格：

- 版本 A：`299 元 / 周期`
- 版本 B：`699 元 / 周期`
- 版本 C：`999 元 / 周期`

### 实验内容

- 指定条件下的项目池筛选
- 重点项目标记
- 周期内若干重点项目的快速判断结果

### 观察指标

- 是否愿意下单
- 周期结束后是否希望续用
- 用户最在意的是“筛选效率”还是“判断结果”

### 判断标准

- 如果用户更看重“你帮我省时间”，说明筛选服务潜力大
- 如果用户更看重“这个项目到底能不能做”，说明单项目报告更强

## 5.3 实验 C：组合打包测试

### 目标

验证单项目报告和筛选服务是否适合单独卖，还是打包卖更容易成交。

### 实验方式

测试三种报价方式：

- 仅单项目报告
- 仅连续筛选服务
- 筛选服务 + 每周期若干重点项目报告

### 判断标准

- 如果组合包转化明显更高，说明用户购买的是“持续决策支持”，而不是单个功能

## 6. 奖励实验

## 6.1 实验 D：奖励有无测试

### 目标

验证没有奖励时是否还有足够专业用户愿意参与。

### 实验方式

将供给侧用户分两组观察：

- A 组：无外部 AI token 奖励，仅有声望/排行榜
- B 组：有小额外部 AI token 奖励

### 观察指标

- 参与率
- 完成率
- 评估质量
- 次周是否继续参与

### 判断标准

- 如果无奖励就没人参与，说明平台早期需要补贴
- 如果有奖励但质量显著下降，说明奖励方向有问题

## 6.2 实验 E：奖励强度测试

### 目标

验证多大强度的奖励才有感知，但又不至于吸来大量低质量参与。

### 实验方式

测试三档奖励：

- 低档：象征性奖励
- 中档：明显有吸引力，但预算可控
- 高档：对用户很有吸引力，但风险较大

这里不预先写死 token 数，而由你根据采购成本折算。

### 观察指标

- 新增参与人数
- 低质量提交比例
- 单次有效提交成本

### 判断标准

- 如果高档奖励只带来更多刷子，不带来更高质量供给，就不成立

## 6.3 实验 F：奖励规则测试

### 目标

验证奖励按什么规则发，才能避免“猜平均值”。

### 候选规则

- 规则 A：最接近聚合结果者奖励
- 规则 B：参与完成度 + 历史稳定度
- 规则 C：参与完成度 + 偏差控制 + 历史稳定度

### 判断重点

- 是否鼓励从众
- 是否鼓励认真评估
- 是否容易被用户理解

### 当前建议

优先测试 `规则 C`，因为它更接近“质量贡献奖励”。

## 7. 奖励预算控制方案

`2.0` 奖励实验必须设硬边界。

### 建议约束

- 设置月度奖励总预算
- 设置单用户月度上限
- 设置单项目奖励上限
- 奖励兑换必须经过记录与对账

### 关键指标

- 单次有效评估奖励成本
- 单活跃评估者奖励成本
- 奖励支出 / 收入比

## 8. 先做什么，不先做什么

## 8.1 应优先做的

- 单项目报告价格测试
- 连续筛选服务价格测试
- 有无奖励测试
- 奖励强度测试

## 8.2 暂时不要做的

- 很复杂的会员折扣体系
- 长周期大额预付费方案
- 高成本现金型激励
- 复杂积分商城

## 9. 实验节奏建议

建议按 3 个阶段推进：

### 阶段 1：支付意愿

先验证：

- 单项目报告是否能卖
- 连续筛选服务是否有人买

### 阶段 2：供给侧激励

再验证：

- 奖励是否能拉动参与
- 奖励是否带来低质量流量

### 阶段 3：单位经济性

最后验证：

- 收入能否覆盖奖励和交付成本
- 哪一种产品模型更接近可持续

## 10. 关键记录表

建议每次实验都记录以下内容：

- 实验名称
- 时间范围
- 对象人群
- 价格或奖励规则
- 曝光人数
- 咨询人数
- 付费人数
- 有效参与人数
- 奖励支出
- 结论

## 11. 当前最重要的实验结论标准

如果 `2.0` 的实验做得对，你最终应该能回答这几个问题：

1. 单项目报告和连续筛选服务，谁更容易成交
2. 什么价格带开始有人真实付钱
3. 奖励是否真能带来高质量供给
4. 单位收入能否覆盖奖励和交付成本
