# PPA 2.0 Product Brief

## 执行摘要

`PPA 1.0` 的核心价值已经被证明: 它不是一个简单的表单系统，而是一套可解释的项目评估引擎，能够把风险、工作量、成本和导出链路组织成完整流程。但它当前主要服务于单一操作者和私有使用场景，尚不适合直接面向大众或企业协作。

`PPA 2.0` 的目标不是立刻做成一个大型社区，也不是先追求平台规模，而是把现有评估引擎升级为一个能够公开运行、能够吸引多人参与、并且能够尽快产生收入的“平台化第一步”。最合适的形态不是纯 `2C` 社区，而是 `2B2C` 的轻平台: `C` 端负责参与公开项目评估、形成群体判断，`B` 端为更高质量的评估结果、筛选结论和决策支持买单。

`PPA 2.0` 的核心产品定义是: 基于公开招标项目的群体评估平台 MVP。平台从公开招标网站获取项目线索，将其转化为可参与的标准化评估任务；用户围绕同一项目给出风险、工作量、成本判断；平台对结果做聚合、分歧分析和可信度处理；最终输出对乙方投标/接单决策有价值的评估报告。该版本优先验证三件事：用户是否愿意参与、群体评估是否有价值、是否有人愿意为结果付费。

## 要解决的问题

### 对用户侧

公开招标网站能提供大量项目机会，但只提供原始信息，不提供“这个项目值不值得做、应该报多少、有哪些隐藏风险”的判断。很多软件公司老板、投标人员、项目经理和技术负责人，只能依赖个人经验快速拍板，容易出现三类问题：

- 错把高风险低利润项目当成机会
- 报价没有基准，过高丢单，过低亏损
- 团队内部对同一个项目判断分歧很大，缺少结构化共识

### 对平台侧

`PPA 1.0` 已经有专业评估能力，但它的输入是私有项目，输出偏向个人或内部使用，缺少可持续增长的公共任务池、参与者机制和商业闭环。也就是说，评估引擎有了，但还没有变成一个可运营、可传播、可收费的平台产品。

## 解决方案

`PPA 2.0` 把公开招标项目转化为标准化评估任务，并围绕“同一个项目，多人给出结构化判断”建立一套轻量平台机制。

用户在平台上的核心体验是：

- 浏览平台整理后的公开招标项目
- 选择感兴趣的项目参与评估
- 基于标准化表单提交风险、工作量、成本判断
- 查看该项目的群体平均值、分歧程度和聚合结果
- 通过排行榜、声望和奖励机制获得持续参与动力

平台对外提供的核心价值是：

- 对单个公开项目形成“群体评估基线”
- 为乙方公司提供更可靠的投标/接单参考
- 让原本分散在个人经验中的判断，变成可聚合、可复用、可售卖的数据资产

## 这版产品为什么不同

`PPA 2.0` 不只是“又一个招标信息网站”，也不只是“把 1.0 放到网上”。它有三个关键差异：

1. 招标平台通常解决“看到项目”，不解决“判断项目”；`PPA 2.0` 解决的是决策问题，而不是信息展示问题。
2. 通用评估工具通常是单人或内部使用；`PPA 2.0` 的价值来自多人围绕同一项目形成结构化共识。
3. 你已经拥有 `PPA 1.0` 的评估引擎、计算逻辑和专业方法论，这意味着 `2.0` 不是从零开始做社区，而是从现成的专业能力向外平台化。

## 服务对象

### 主要参与用户

- 产品经理、项目经理、架构师、开发负责人、测试负责人
- 有投标经验或软件交付经验的从业者
- 对项目估算、风险判断、软件报价感兴趣的专业用户

这部分用户不一定直接付费，但他们决定平台是否有评估供给、是否能形成可信结果。

### 主要付费用户

- 中小软件公司老板
- 外包团队或项目承接团队负责人
- 投标人员和售前方案人员
- 需要快速判断“要不要投”“值不值得做”“大概该怎么报”的团队

这部分用户不一定会高频参与评估，但他们最可能为“结果”和“决策依据”付费。

## 商业化路径

`PPA 2.0` 不应以“先做大社区，再慢慢变现”为主线，而应以“先形成可收费的决策支持能力”为优先。

建议的基础商业化结构：

- 免费层：
  - 浏览部分公开项目
  - 参与评估
  - 查看基础聚合结果
  - 获取基础排行榜和个人声望
- 付费层：
  - 查看完整项目评估报告
  - 获取项目可投性判断与风险说明
  - 查看群体分歧与可信度信息
  - 使用筛选、收藏、跟踪、导出等决策工具

优先收费对象不是泛用户，而是需要为接单决策负责的人。

## 首个切口建议

为了符合你当前预算、精力和快速变现诉求，`PPA 2.0` 不应一开始覆盖所有公开招标项目，而应先切一个更窄但更容易成交的场景。

建议首个切口为：

- 只聚焦 `软件开发 / 信息化 / 数字化建设` 相关招标项目
- 优先覆盖你已经能够稳定采集和理解的数据源
- 优先服务“中小软件公司 / 外包团队 / 投标负责人”的接单决策

建议首个可卖产品不是年费大订阅，而是更容易成交的轻量付费项：

- 单项目深度评估报告
- 项目可投性判断
- 群体评估结果导出
- 连续项目机会筛选服务

当前版本决策：

- 首批项目池明确只覆盖 `软件开发 / 信息化 / 数字化建设` 相关招标
- 首个收费单元不做二选一，而是并行验证：
  - `单项目评估报告`
  - `连续项目筛选服务`

这意味着 `2.0` 的启动原则应该是：

- 先做窄场景高价值，不做大而全流量平台
- 先验证“有人愿意为结果付费”，再扩大用户规模
- 先把数据质量和评估可信度做出来，再放大奖励机制

## 激励机制原则

你提出的“购买 AI 平台 token 作为奖励”是可行方向，但 `2.0` 不应把它设计成高成本、可提现、强金融属性的激励系统，而应先作为低成本、可控的运营激励。

`2.0` 建议采用的奖励原则：

- 奖励以 `AI token / AI 使用额度 / 平台权益` 为主，不直接发现金
- 奖励允许兑换外部 AI 平台的实际可用额度或 token，以提升吸引力
- 奖励目标是促进高质量参与，而不是单纯刷次数
- 不应只用“最接近平均值”作为唯一奖励依据，否则会鼓励从众，而不是鼓励专业判断
- 第一阶段可以采用“参与完成度 + 结果稳定性 + 历史可信度”组合评分

因此，`2.0` 的奖励设计建议是：

- 前台对用户表达为“可兑换真实 AI 使用权益”
- 后台对平台控制为“额度池 + 兑换门槛 + 风控限制”
- 奖励发放频率应低于参与频率，避免把平台做成纯薅羊毛场

## 2.0 范围

### 2.0 必须做的

- 从公开招标网站导入并清洗项目池
- 将招标项目转化为统一评估任务
- 提供面向多人参与的标准化评估表单
- 输出项目聚合结果：平均值、区间、分歧度、基础可信度
- 提供用户账户、排行榜、基础声望体系
- 提供可控的 `AI token` 奖励机制
- 生成可售卖的项目评估报告
- 提供连续项目筛选与跟踪能力
- 提供最基础的运营后台

### 2.0 明确不做的

- 直接现金奖励或复杂结算系统
- 泛社区化讨论产品
- 完整专家市场或评审交易平台
- 面向所有行业的全量招标覆盖
- 大而全的企业协同、组织权限和复杂订阅体系

## 成功标准

`PPA 2.0` 成功与否，不看功能数量，而看是否跑通以下三条链路：

1. **供给链路**：公开招标项目能否持续进入平台并形成可参与任务
2. **参与链路**：是否有足够专业用户愿意持续参与评估
3. **付费链路**：是否有团队愿意为评估结果和决策支持买单

建议首阶段观察指标：

- 平台能够稳定发布可评估项目
- 出现一批持续参与评估的核心用户
- 单个项目能够获得足够数量的有效评估样本
- 至少形成首批真实付费用户或付费报告订单
- 平台固定运维成本可控制在你当前预算边界内

## 版本策略

`PPA 2.0` 的本质不是“终极平台版”，而是“验证平台化是否成立”的版本。它要解决的是：专业评估引擎能否变成公开任务、群体共识和付费结果。

如果 `2.0` 跑通，`PPA 3.0` 再进入以下方向会更稳：

- 更成熟的奖励与反作弊机制
- 更强的专家评审体系
- 更完整的企业订阅与报告产品
- 更稳定的网络效应与社区结构

## 一句话版本定义

`PPA 2.0` 是一个基于公开招标项目的群体评估平台 MVP，用低成本平台机制把专业项目评估能力转化为可参与、可聚合、可售卖的决策支持产品。
