# PPA 2.0 PRD Decision Log

This file records PRD decisions for `docs/prd2.0/prd.md`.

## 2026-06-24

### D-001: Use `docs/prd2.0/prd.md` as the integrated V2 PRD

**Decision:** Keep the existing V2 PRD as the master PRD and add a direction map instead of creating an unrelated standalone PRD.

**Reason:** The existing PRD already contains the SaaS/platform direction. The current work needs to integrate database migration into the V2 plan, not fork the planning surface.

**Captured in:** `docs/prd2.0/prd.md` section `0`.

### D-002: Split V2 PRD into three directions

**Decision:** Structure V2 around three directions:

1. Data and storage foundation migration.
2. SaaS platform planning and supplementation.
3. Integration of other `docs/prd2.0/` business, investor, pricing, and reward documents.

**Reason:** The user explicitly asked for at least these three directions, and direction one is the current implementation priority.

**Captured in:** `docs/prd2.0/prd.md` section `0.1`.

### D-003: Treat direction one as data and storage foundation migration

**Decision:** Direction one is broader than database-only migration. It includes SQLite to Supabase PostgreSQL, project attachments to Supabase Storage, Git/file hygiene, retention policy, verification, and rollback.

**Reason:** The decision documents show that project attachments, crawler data, AI logs, backups, and database migration are coupled through the same local-file and online-service constraint.

**Captured in:** `docs/prd2.0/prd.md` section `A`.

### D-004: Use Supabase hosted PostgreSQL and Storage as the selected path

**Decision:** The PRD uses “Supabase hosted PostgreSQL” for the database and “Supabase Storage” for P0 project attachments.

**Reason:** The local decision documents already selected Supabase for low-cost online database, backup, storage, and V2 extensibility. The PRD keeps Auth/Realtime/Edge Functions as future capabilities, not direction-one scope.

**Captured in:** `docs/prd2.0/prd.md` sections `0.3`, `A.6`, and `A.7`.

### D-005: Correct Supabase capacity wording before implementation

**Decision:** The PRD flags Supabase capacity as an implementation-time verification item because the local decision documents conflict.

**Reason:** Local docs mention both `500 MB` and `10 GB` free capacity. Official Supabase documentation checked on 2026-06-24 showed Free plan limits of `500 MB Database Size`, `1 GB Storage Size`, and `2` free projects. Source: https://supabase.com/docs/guides/platform/org-based-billing

**Captured in:** `docs/prd2.0/prd.md` sections `0.3`, `A.9`, and `A.10`.

### D-006: Keep SaaS Auth and multi-tenant decisions out of direction one

**Decision:** Supabase Auth, multi-tenant organization structure, payment, membership, and SaaS permissions remain direction-two work.

**Reason:** Direction one is infrastructure enablement. Pulling SaaS identity and payment into this phase would expand scope before database and storage safety is resolved.

**Captured in:** `docs/prd2.0/prd.md` sections `A.2`, `A.7`, and `A.10`.

### D-007: Make `prd.md` the single index for `docs/prd2.0/`

**Decision:** Use `docs/prd2.0/prd.md` as the only directory-level index for the folder. Keep `README.md` as a light entry point, not a second source of truth.

**Reason:** The folder has accumulated product, business, investor, and pitch materials. A single index reduces navigation drift and keeps future updates anchored to the PRD.

**Captured in:** `docs/prd2.0/prd.md` section `0.4`; `docs/prd2.0/README.md`.
