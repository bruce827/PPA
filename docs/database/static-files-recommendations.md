### 2.1 数据库（73 MB）🔴 P0

#### 当前状态

- **大小**: 73.09 MB
- **表数**: 26 张
- **数据行数**: 约 2,800 行
- **Git 历史**: 已提交 2 次
- **问题**: 包含敏感数据，不应提交到代码库

#### 问题分析

**为什么 73 MB 偏大？**

| 原因 | 占用空间 | 说明 |
|------|---------|------|
| AI 评估日志 | ~30 MB | 769 行，包含完整请求/响应 |
| 招标项目 Staging | ~20 MB | 929 行，包含正文内容 |
| 项目详情 JSON | ~10 MB | 53 行，平均 200 KB/行 |
| 其他表 | ~13 MB | 索引、配置等 |

**平均每行 26 KB**（正常应 < 1 KB），说明存在大字段存储。

#### ✅ 建议

**方案**: 迁移到 **Supabase PostgreSQL**

**理由**:
- ✅ **消除 Git 安全风险**（数据库不再在代码库）
- ✅ **自动备份**（Supabase 每日备份）
- ✅ **在线访问**（支持 V2 Agent 接口）
- ✅ **零成本**（500 MB 免费层，当前 40 MB 足够）
- ✅ **数据完整性**（100% 迁移，零丢失）

**详细方案**: 见 `postgresql-migration-plan.md`

**工作量**: 2-3 小时

| 特性 | 当前（本地） | Supabase Storage |
|------|------------|----------------|
| **上传/下载** | 手动管理 | ✅ 开箱即用 |

| **权限控制** | ❌ 无 | ✅ 签名链接（1小时过期） |
| **在线可用** | ❌ 本地 only | ✅ 云端存储 |
| **成本** | ✅ 免费 | ✅ 免费（500 MB 内） |

**迁移后架构**:
```
用户上传附件
  → 后端接收文件
  → 存储到 Supabase Storage（bucket: project-attachments）
  → 数据库保存文件路径
  → 后端生成签名链接（1小时有效）
  → 前端通过签名链接下载
```

**工作量**: 2-3 小时

  - 使用 `git-filter-repo` 移除所有历史中的 db 文件
  - 一次性操作，15 分钟
  - 不影响本地备份

**决策理由**:
- 两者解决不同问题（互补，不是互斥）
- 备份-恢复：迁移失败的应急方案
- 清理历史：消除安全风险 + 减小仓库

**备份步骤**:
```bash
BACKUP_TIME=$(date +%Y%m%d-%H%M%S)
cp server/ppa.db "server/backups/ppa.db.pre-pg-migration-${BACKUP_TIME}"
sqlite3 "server/backups/ppa.db.pre-pg-migration-${BACKUP_TIME}" "SELECT COUNT(*) FROM projects;"
```

**恢复步骤**:
```bash
cp "server/backups/ppa.db.pre-pg-migration-${BACKUP_TIME}" server/ppa.db
sed -i '' 's/DB_TYPE=postgres/DB_TYPE=sqlite/' server/.env
node server/index.js
```

**清理步骤**: 见 `docs/database/git-history-cleanup.md`
