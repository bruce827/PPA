## 1. 背景

现有“项目机会 -> 招标网站”模块已经完成首版建设，数据库中已有首批基础站点数据。

现在新增了一份来源更广的新数据源 `docs/招标网站.xml`。这份数据源覆盖了：

- 全国公共平台
- 央采军采平台
- 省市自治区政采与公共资源平台
- 企业招标采购平台
- 第三方交易平台

但该数据源同时存在以下问题：

- 不是标准 XML，而是 HTML 片段
- 存在重复链接和重复名称
- 存在 `http/https`、尾斜杠、命名差异
- 混入了登录页、审批监管平台、电子卖场、网上商城、服务超市等不适合直接进入主库的入口
- 企业平台和第三方平台口径较宽，不能直接批量导入

因此后续工作不能直接“全量导入”，而必须分阶段执行。

## 3. 目标

本轮数据补充与清洗的目标不是“把 XML 全部导进去”，而是完成以下三件事：

1. 在不破坏现有库稳定性的前提下，安全补充高价值主站点
2. 识别并归并现有库中与新源重复但 URL 形态不同的记录
3. 为企业平台、第三方平台建立明确的导入口径，避免主库失控

## 4. 总体策略

执行策略采用“三段式”：

### 第一段：安全补充

先处理风险最低、价值最高的主站类记录：

- 全国公共平台
- 央采军采平台
- 各省公共资源交易主站
- 各省政府采购主站

原则：

- 只处理 `keep` 类记录
- 不碰 `review`
- 不碰 `exclude`

### 第二段：归并修正

再处理现有库和新源之间的“同站不同写法”问题：

- `http/https` 不同
- 尾斜杠差异
- 名称不同但站点相同
- 旧名称与新名称冲突

原则：

- 优先修正名称与别名
- URL 是否替换为新源版本要谨慎处理
- 不做强制批量覆盖

### 第三段：边界源审查

最后才处理：

- 企业招投标平台
- 企业采购平台
- 第三方交易平台
- 云平台类入口

原则：

- 先定口径，再导数据
- 不允许一边审、一边批量入库

## 5. 执行阶段

## 阶段 0：冻结底稿

目标：

- 以 `docs/csv/招标网站-xml-结构化清单.csv` 作为后续操作唯一底稿

- 当前整理说明文档

完成标准：

- 后续所有补充、清洗、审查，都基于 CSV 做标注

## 阶段 1：安全补充候选清单

目标：

- 从 `keep` 中筛出可优先补库的高价值候选

范围：

- 全国公共平台
- 央采军采平台
- 各省公共资源/政府采购主站

具体动作：

1. 对 `keep` 记录做柔性去重分组
2. 与现有库比对，拆成四类：
   - `already_covered`
   - `name_update_candidate`
   - `url_variant_candidate`
   - `new_candidate`
3. 对 `new_candidate` 建立拟导入清单

说明：

- 这一阶段只形成“候选导入集”，不直接导库
- `docs/csv/招标网站-xml-直接补充-dry-run.csv` 作为后续补库的 dry-run 底稿
- `docs/csv/招标网站-xml-直接补充-同名待确认.csv` 里的 7 条不能直接新增

预期结果：

- 84 条 `keep` 里，预计有一部分会与现有库重合
- 真正新增数量大概率显著少于 84
- 结合第二轮差异分析，当前“可直接补充主站”已收缩到 `35` 条
- 结合 dry-run，真正可先按“拟新增”推进的只剩 `28` 条

## 阶段 2：现有库归并策略

目标：

- 明确如何处理当前库和新源之间的重复站点

重点问题：

1. 名称是否更新
2. `alias_name` 是否补充
3. URL 是否保留旧值还是替换新值
4. 是否需要记录“发现于 XML 新源”

建议规则：

- 若新源名称更规范，更新主名称，旧名称写入 `alias_name`
- 若仅 `http/https` 或尾斜杠不同，默认不新建记录
- 若新源 URL 明显更优且可访问性更高，再考虑替换旧 URL
- 若新源只是云平台、商城、登录入口，不覆盖主站 URL

产出：

- 一份“现有库更新建议表”
- 当前已形成 `docs/csv/招标网站-xml-疑似归并清单.csv`
- `same_name_review` 这 7 条可优先并入此阶段处理
- 当前已形成 `docs/csv/招标网站-xml-同名归并建议.csv`

## 阶段 3：企业平台分层审查

目标：

- 解决企业平台口径过宽的问题

当前分组：

- 企业招投标平台：48
- 企业采购/电子商务平台：62

建议再细分为：

- `企业招投标主平台`
- `企业采购交易平台`
- `企业电子商务/商城`
- `企业内部或登录型入口`

建议导入原则：

- 明确包含“招标、投标、交易、采购公告”的企业主平台，优先保留
- 纯商城、采购中台、电子商务平台默认不优先导入
- 带明显登录路径的入口默认不导入主库

产出：

- 一份企业平台审查清单
- 当前已形成 `docs/csv/招标网站-xml-review-二轮分组.csv` 中的企业相关子集

## 阶段 4：第三方平台审查

目标：

- 对 107 条第三方平台建立更清晰的准入规则

风险：

- 第三方平台数量多
- 质量参差不齐
- 很容易把主库变成“泛招标网址大全”

建议口径：

- 保留行业内高识别度、覆盖面广的平台
- 弱平台、跳转平台、纯营销站默认不优先导入
- 若未来要扩“项目机会”其他子模块，可把第三方平台拆到单独子类处理

产出：

- 一份第三方平台白名单候选集
- 当前已形成 `docs/csv/招标网站-xml-review-二轮分组.csv` 中的第三方平台待审子集

## 阶段 5：最终导入与清洗执行

目标：

- 在前面 4 个阶段都确认后，再做正式数据库操作

执行顺序建议：

1. 新增高置信度主站
2. 更新现有记录名称/别名
3. 审慎处理 URL 替换
4. 再导入审查通过的企业平台
5. 最后再决定第三方平台是否进入主库

执行要求：

- 必须先产出 dry-run 清单
- 导入前后都要统计记录数
- 所有新增/更新必须可追溯

## 6. 当前建议的实际执行顺序

如果后续进入真正执行，建议严格按下面顺序推进：

1. 先以 `docs/csv/招标网站-xml-直接补充-dry-run.csv` 为准，只看 `28` 条 `create_candidate`。
2. `docs/csv/招标网站-xml-直接补充-同名待确认.csv` 这 `7` 条不要直接新增，应先按归并思路处理。
3. 这 `7` 条里，先把 `candidate_update_primary_url` 的 1 条并入最终 dry-run 变更表；其余 6 条保持“先验证再改”。
4. 现在可直接以 `docs/csv/招标网站-xml-最终dry-run变更表.csv` 作为第一批执行底稿。
5. `docs/csv/招标网站-xml-待在线校验后再变更.csv` 这 6 条维持隔离，先校验再决定是否改主 URL。
6. 再处理 `docs/csv/招标网站-xml-疑似归并清单.csv` 这 `51` 条，只做名称、别名、覆盖关系判定，不急着替换 URL。
7. 单独审 `cloud_platform_variant` 和 `enterprise_bidding_platform`，这是最容易口径漂移的两组。
8. `enterprise_procurement_platform` 与 `third_party_trading_platform` 暂时不进主库，除非先补充一份明确准入标准。

## 6. 推荐的实施批次

建议按批次推进，不要一次性处理全部 331 条：

### 批次 A：主站补充

- 范围：48 条企业招投标类

### 批次 D：企业采购与第三方平台

- 风险：高

## 7. 风险点

### 风险 1：误把商城和登录页导入主库

应对：

- `exclude` 类默认不进入导入候选集

### 风险 2：因协议差异造成重复站点

应对：

- 同时使用严格匹配和柔性匹配

### 风险 3：企业平台口径失控

应对：

- 先分层，后导入

### 风险 4：第三方平台过多导致主库噪声上升

应对：

- 建白名单，不做全量纳入

## 8. 建议的下一步

最合理的下一步不是导库，而是继续做“候选层清洗”。

建议先执行：

1. 先按 `docs/csv/招标网站-xml-最终dry-run变更表.csv` 处理第一批 `29` 条
2. 再对 `docs/csv/招标网站-xml-待在线校验后再变更.csv` 这 `6` 条补充真实校验结果
3. 然后再单独进入企业平台与第三方平台审查

只有当这三份清单都确认后，才进入数据库补充和清洗执行。

## 9. 当前结论

这个新数据源有价值，但不能直接导入。

更合适的使用方式是：

- 先作为“补充候选源”
- 先补主站
- 再做归并
- 最后再审企业与第三方平台

当前最稳妥的执行路径是：

- 先做 `keep` 候选清单
- 再做现有库归并建议
- 暂不做数据库写入
