# 小程序会员功能 MVP 已知延期风险

## 文档目的

这份文档用于汇总当前会员付费小程序在 `Epic 1 - Epic 5` 已实现后，明确已知但暂不处理的风险项。

这些风险当前被视为 **MVP 验证阶段可接受延期项**，不是这一轮必须马上修复的内容。

## 当前结论

1. 当前版本仍适合内部验证和小范围试跑
2. 当前版本不建议在未重审这些风险前，直接进入更大规模真实付费放量
3. 后续只要满足以下任一条件，就应优先回头处理这些延期项：
   - 开始持续接入真实付费用户
   - 单账号历史订单、会员记录、操作日志明显累积
   - 人工评估关联开始被频繁增量编辑
   - 多套餐从“展示占位”切成真实可售
   - 经营分析开始被当作正式业务判断依据

## 按 Epic 汇总

### Epic 1: 会员入口与受限浏览体验

当前没有独立保留的高/中优先级延期风险。

说明：

1. 登录、列表、详情门禁和会员三态在当前验证阶段已经闭环
2. 但它仍会受到下文“跨 Epic 共用查询窗口限制”的间接影响

### Epic 2: 会员开通、续费与权限生效闭环

#### 风险 1：下单缺少未完成订单幂等保护

- 当前状态：未修复
- 风险级别：高
- 影响范围：`Story 2.1`，并影响 `Story 2.2 / 2.3 / 2.4`

说明：

1. 同一用户重复点击“立即开通”或“立即续费”时，服务端目前会继续新建订单
2. 当前没有“同一用户存在未完成订单时优先复用”的服务端幂等规则
3. 在弱网、双击、重复重试或多端并发时，可能产生多笔未完成订单

当前为什么可以延期：

1. 当前仍处于内部验证阶段
2. 当前没有真实支付商户闭环，重复扣费风险尚未转化为真实财务损失
3. 当前仍可通过人工筛查订单控制误差

后续什么时候必须处理：

1. 开始真实微信支付联调前
2. 或开始允许外部真实用户自行付费前

相关代码：

- `frontend/ppa_miniapp/cloudfunctions/createMembershipOrder/index.js`

#### 风险 2：会员与订单“最新记录”查询使用固定窗口后再内存排序

- 当前状态：未修复
- 风险级别：中
- 影响范围：`Story 2.2 / 2.3 / 2.4`

说明：

1. 当前多个查询会先取固定数量记录，再在内存中排序挑“最新”
2. 当同一 `openid` 下的订单/会员记录超过窗口数后，真正最新记录可能根本没被取到
3. 这会导致会员状态、订单刷新、诊断结果逐步偏离真实最新状态

当前为什么可以延期：

1. 当前目标仍是 very small closed beta
2. 单用户短期内不太可能积累到足够多的会员/订单历史

后续什么时候必须处理：

1. 单个测试用户历史订单明显增多时
2. 开始有稳定续费用户时
3. 开始依赖这些结果做正式运营判断时

相关代码：

- `frontend/ppa_miniapp/cloudfunctions/getMembershipStatus/index.js`
- `frontend/ppa_miniapp/cloudfunctions/getTenderDetail/index.js`
- `frontend/ppa_miniapp/cloudfunctions/queryMembershipDiagnostics/index.js`

### Epic 3: 运营排查与异常兜底能力

当前没有新增的独立高/中优先级延期风险，但会继承 `Epic 2` 中“最新记录查询窗口限制”的影响。

说明：

1. 当前内部权限、事务留痕和 `manual_fixed` 语义已经补齐
2. 但如果订单、会员、操作日志历史继续增长，诊断函数仍可能读取不到真正最新的记录

### Epic 4: 会员留存与经营增强

#### 风险 3：经营分析会静默截断数据

- 当前状态：未修复
- 风险级别：中
- 影响范围：`Story 4.4`

说明：

1. 当前经营分析函数会对订单、会员、访问日志读取设置固定上限
2. 当数据超过上限时，页面不会提示“统计已截断”
3. 付费数、续费数、访问数和高关注项目结果会被系统性低估

当前为什么可以延期：

1. 当前经营分析主要用于 MVP 阶段的方向判断，不是正式 BI 报表
2. 当前数据规模小，短期内截断影响有限

后续什么时候必须处理：

1. 经营分析开始作为正式运营决策依据时
2. 活跃用户和访问日志明显增长时

相关代码：

- `frontend/ppa_miniapp/cloudfunctions/getMembershipInsights/index.js`

#### 风险 4：多套餐已经有配置骨架，但生效时长仍固定为 30 天

- 当前状态：未修复
- 风险级别：中
- 影响范围：`Story 4.3`，并影响 `Epic 2` 的续费/开通逻辑

说明：

1. 当前月卡、季卡、年卡配置骨架已经存在
2. 但会员生效和人工补开逻辑仍按固定 `30 天` 计算
3. 一旦把季卡或年卡从“敬请期待”切成真实可售，会直接发错权益时长

当前为什么可以延期：

1. 当前只有月会员开放
2. 季卡和年卡仍只是展示占位，不影响现阶段真实用户

后续什么时候必须处理：

1. 任意非月卡套餐准备切换为 `enabled` 前

相关代码：

- `frontend/ppa_miniapp/cloudfunctions/createMembershipOrder/index.js`
- `frontend/ppa_miniapp/cloudfunctions/refreshMembershipOrder/index.js`
- `frontend/ppa_miniapp/cloudfunctions/fixMembershipOrder/index.js`

### Epic 5: 人工评估升级与 Web 能力衔接

#### 风险 5：人工评估关联写入是整条覆盖，不是 merge 更新

- 当前状态：未修复
- 风险级别：高
- 影响范围：`Story 5.1 / 5.2 / 5.3 / 5.4`

说明：

1. 当前内部运营如果二次调用评估关联写入函数，只补部分字段
2. 旧的正文、摘要、风险、建议动作或成果物字段可能被空值覆盖
3. 这属于真实数据覆盖风险，不是单纯展示误差

当前为什么可以延期：

1. 当前人工评估仍处于早期试验接入
2. 当前写入频率低，可以通过“每次全量回写”人为规避

后续什么时候必须处理：

1. 开始频繁增量维护评估结果时
2. 不同人分批补正文、成果物、服务导流字段时
3. 评估记录开始成为正式业务资产时

当前临时规避方式：

1. 内部调用 `upsertTenderEvaluationLink` 时，默认按“整条全量回写”执行
2. 不要只补 `artifacts` 或 `service_*` 字段而不带旧正文

相关代码：

- `frontend/ppa_miniapp/cloudfunctions/upsertTenderEvaluationLink/index.js`

## 跨 Epic 风险总览

### 当前真正延期但未改动的 5 项

1. 订单缺少未完成订单幂等/去重保护
2. 会员/订单/诊断“最新记录”读取存在固定窗口限制
3. 经营分析存在静默截断
4. 多套餐时长规则尚未真正贯通
5. 人工评估关联是整条覆盖而非 merge 更新

## 当前阶段建议

### 当前可以继续做的事

1. 继续用当前版本做内部验证
2. 继续按会员门禁、手工开通、续费、人工评估、成果物和 Web 导流链路做用户试跑
3. 继续在小范围内验证“20 元/月是否有价值”

### 当前不建议直接做的事

1. 在未补订单幂等前，直接对外放开真实多用户支付
2. 在未补 merge 更新前，让多人并行维护人工评估记录
3. 在未补多套餐时长规则前，直接开放季卡或年卡
4. 在未补统计截断说明前，把经营分析结果当正式经营报表使用

## 与无商户测试的关系

如果你当前还没有开通微信支付商户，建议同步参考：

- `docs/miniapp-bidding/membership-manual-testing-without-merchant.md`

当前阶段最稳的策略是：

1. 保持真实架构不变
2. 不新增“假支付成功”前台逻辑
3. 用手工订单 + 手工会员状态继续做闭环验证
4. 等商户能力具备后，再补真实支付与回调联调
