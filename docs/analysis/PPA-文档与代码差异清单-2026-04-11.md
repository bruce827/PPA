# PPA 文档与代码差异清单

- 产出时间：2026-04-11
- 审计目标：识别 `docs/prd` 与当前实现之间会误导开发、测试、报价或产品判断的偏差
- 审计原则：以代码实现为准，文档按“是否仍能安全指导行动”来评估

## 1. 结论先看

当前最值得警惕的不是“少几份文档”，而是文档库里同时存在：

- 旧口径
- 新口径
- 已实现但仍标 Draft 的能力
- README 中被标成删除但仓库里仍保留的文件

因此，`docs/prd` 目前更像“现行规格 + 历史遗留 + 规划草案”的混合区，而不是严格的一致性文档库。

## 2. 高优先级差异

## 2.1 风险因子公式文档已过时

文档写法：

- `docs/prd/assessment-spec.md`
- `docs/prd/calculation-logic-spec.md`

这些文档仍在使用类似 `riskScore / 100` 的线性公式来描述风险因子。

代码现状：

- `server/utils/rating.js`
- `frontend/ppa_frontend/src/utils/rating.ts`

当前实现已经改为：

- 先根据已配置风险项的最大可得分计算分母
- 再按风险得分占比进行分段插值
- 最终进行上限封顶

影响：

- 产品、测试、业务人员如果按旧文档验算，会得到错误结果
- 新功能若继续引用旧公式，会引入二次漂移
- 培训和交接时容易把“旧逻辑”误当现行逻辑

建议：

- 将旧公式文档明确标注为过期
- 单独补一份“风险因子口径说明”，并把代码路径写进去
- 在总览文档中只保留一个唯一公式口径

## 2.2 风险总分来源范围文档不完整

文档写法：

- 多数评估与计算文档主要围绕配置型风险项展开

代码现状：

- `server/services/calculationService.js`

当前代码在计算总风险分时，除了配置风险项，还会纳入：

- `ai_unmatched_risks`
- `custom_risk_items`

影响：

- 文档读者会低估总风险分
- 风险因子、总成本和解释文本都可能被误判
- 测试数据如果只覆盖配置项，会漏掉真实分支

建议：

- 把风险得分来源统一拆成三类写清楚
- 在评估文档和计算文档里都补上额外风险来源
- 测试用例按三类来源分别覆盖

## 2.3 角色单价单位口径冲突

文档写法：

- `docs/prd/config-spec.md` 把 `unit_price` 写成“元/月”

代码现状：

- `frontend/ppa_frontend/src/pages/Config.tsx` 明确展示为“元/人天”
- `server/services/calculationService.js` 按“工作天数 x 单价”计算

影响：

- 这是会直接影响报价理解的口径冲突
- 业务沟通、导入数据和测试样例可能全部错位
- 如果有人按月单价录入，结果会整体失真

建议：

- 立即修正 `config-spec.md`
- 对所有出现 `unit_price` 的文档做一次全文检索
- 后续统一使用“元/人天”或“元/人/天”其中一种写法，不再混用

## 3. 中优先级差异

## 3.1 API 返回结构并未真正完全统一

文档写法：

- 总体架构说明倾向于描述“所有接口都返回 `{ success, data, error }`”

代码现状：

- `server/controllers/calculationController.js`
- `server/controllers/projectController.js`

这部分接口仍大量返回：

- `{ data: ... }`
- 或直接返回结果对象

同时，AI、机会域等接口又更接近 `{ success: true, data }`。

影响：

- 前后端联调时容易基于错误假设封装接口
- 新接口开发者会不知道应该对齐哪一套
- 文档对外说明会显得比实现更整齐

建议：

- 二选一：要么统一代码，要么下调文档承诺
- 如果短期不统一，至少补一份“当前响应格式现状说明”

## 3.2 `docs/prd/README.md` 不能再当状态真相

文档写法：

- `README.md` 中部分能力被标为已完成
- 同时又把部分文档标记为已删除

代码与仓库现状：

- 某些被标已完成的能力，其独立 PRD 仍是 Draft
- 某些被标已删除的文件仍存在于目录中

影响：

- 阅读顺序会被带偏
- 新成员容易把旧文档当现行规范
- 产品规划、需求盘点和验收清单都可能基于错误状态

建议：

- 把 `README.md` 降级成目录索引
- 新增字段区分 `active / draft / archived`
- 物理移动或归档已废弃文档，避免继续与现行文档混放

## 3.3 多个能力已经落地，但文档仍停留在草案语气

典型区域：

- AI 项目标签与相关业绩能力
- CSV 业绩库
- 项目机会域中的去重、归档、字段解析等能力

相关代码：

- `server/routes/ai.js`
- `server/models/projectModel.js`
- `server/routes/contracts.js`
- `frontend/ppa_frontend/src/pages/Assessment/Contracts.tsx`
- `server/routes/opportunity.js`

影响：

- 容易让人误判系统成熟度
- 需求讨论时可能重复定义已存在能力
- PRD 会变成“像规划、但实际已实现”的失真说明

建议：

- 对这几块分别补“实现现状”段落
- 把 Draft 文档里已经落地的部分拆成“已实现 / 待扩展”

## 4. 低到中优先级差异

## 4.1 推送链路仍有配置外硬编码痕迹

代码现状：

- `server/services/pushService.js`

当前推送快照构建中仍可见固定差旅成本值的历史兼容逻辑，例如 `7500`。

影响：

- 推送结果与主计算链的配置驱动原则不完全一致
- 后续如果配置变更，推送侧可能悄悄漂移

建议：

- 将推送侧差旅成本统一改为复用配置模型或计算服务
- 把“历史兼容逻辑”显式注释并列入后续清理项

## 5. 当前最需要修的不是哪份文档

如果只能先做一轮治理，我建议优先顺序如下：

1. 修风险因子与风险总分口径
2. 修角色单价单位
3. 处理 README 的状态漂移
4. 给已落地但仍像草案的能力补“实现现状”
5. 决定是否统一 API 响应结构

## 6. 建议的文档治理动作

为了避免这个问题反复出现，建议后续采用以下治理动作：

1. 为每份 PRD 增加状态字段：`active`、`draft`、`archived`
2. 在总览文档中固定“唯一有效公式”和“唯一有效单位”
3. 为关键口径单独建短文档，不把公式散落在多个 PRD 中
4. 每次较大实现变更后，至少同步更新 `project-overview.md`
5. 将历史或废弃文档迁移到单独的 `archive` 目录

这份差异清单的目标不是挑错，而是把最容易误导后续工作的偏差先钉住，避免团队继续在旧口径上叠加新实现。
