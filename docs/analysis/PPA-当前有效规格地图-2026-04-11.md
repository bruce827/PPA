# PPA 当前有效规格地图

- 产出时间：2026-04-11
- 适用范围：`PPA` 主仓库当前已实现能力
- 目的：给后续开发、补文档、验收和需求讨论提供一个“当前事实源地图”

## 1. 这份地图怎么用

PPA 现在不是“单个 PRD 驱动的单产品”，而是一个由多个能力域拼出来的系统。要避免误判，当前事实源优先级建议固定为：

1. `代码实现`
2. `docs/prd/project-overview.md`
3. 各模块 PRD
4. `docs/prd/README.md`

其中：

- `代码实现` 决定“系统今天实际上怎么运行”
- `project-overview.md` 最接近当前总口径
- 分模块 PRD 可作为局部背景，但存在新旧口径混杂
- `README.md` 目前更适合作为索引，不适合作为单一事实源

## 2. 系统主干

当前主链路可以概括为：

`参数配置 -> 分步评估 -> 后端统一计算 -> 项目沉淀 -> 导出/商务报价/推送`

这里的几个关键判断：

- 标准评估和 Web3D 评估共用“项目沉淀、历史、导出、推送”的骨架，但评估步骤和计算细节不同
- `projects` 已经不只是基础项目表，而是项目事实快照中心
- 商务报价是后置加工层，不应被理解成实施成本本体
- 风险、工作量、成本的最终计算口径已经收敛到后端，而不是前端临时计算

## 3. 当前已验证的能力地图

## 3.1 标准项目评估

这是当前系统的主业务域，已形成完整闭环。

- 前端入口：`frontend/ppa_frontend/.umirc.ts`
- 主要页面：
  - `src/pages/Assessment/New.tsx`
  - `src/pages/Assessment/History.tsx`
  - `src/pages/Assessment/Detail.tsx`
  - `src/pages/Assessment/Contracts.tsx`
- 后端主链：
  - `server/routes/projects.js`
  - `server/controllers/projectController.js`
  - `server/services/projectService.js`
  - `server/services/calculationService.js`
  - `server/models/projectModel.js`

当前有效认知：

- 评估结果会以 JSON 快照方式沉淀到 `assessment_details_json`
- 项目已经区分 `standard` 和 `web3d`
- 项目还附带 `tags_json`、`business_quote_json` 等后续加工信息
- 历史页已经不是简单列表，而是“查看、导出、商务报价状态、推送”的操作中心

## 3.2 计算与参数配置

这一层是整个系统的规则中心。

- 前端入口：
  - `frontend/ppa_frontend/src/pages/Config.tsx`
- 后端配置与计算：
  - `server/routes/config.js`
  - `server/controllers/configController.js`
  - `server/services/calculationService.js`
  - `server/utils/rating.js`
  - `server/models/configModel.js`

已验证存在的配置项：

- 角色与单价
- 风险项与评分选项
- 差旅成本
- 商务报价配置
- Web3D 风险配置入口

当前有效认知：

- 成本计算已经明确依赖角色单价、工作量、差旅、风险因子
- 风险因子不再是线性公式，而是按风险得分占最大分值比例进行分段插值，并封顶
- 前端也保留了一套同口径工具函数，但应视为界面联动，不应取代后端作为最终裁决

## 3.3 AI 能力层

AI 在当前系统里已经是一个能力簇，不再只是“风险识别按钮”。

- 路由入口：`server/routes/ai.js`
- 配置与模板：
  - `server/models/aiModelModel.js`
  - `server/routes/modelConfig.js`
  - `frontend/ppa_frontend/src/pages/ModelConfig/`
- 运行日志：
  - `server/services/aiFileLogger.js`
  - `frontend/ppa_frontend/src/pages/Monitoring/`

已验证能力：

- AI 风险评估
- 风险名称标准化
- 项目模块分析
- 工作量评估
- 项目标签生成
- 多模型配置
- Prompt 模板管理
- AI 日志监控

当前有效认知：

- AI 已经参与“识别风险、拆模块、估工作量、打标签”等多个环节
- 模型和 Prompt 已经产品化，不再是代码里的静态常量
- 文件日志和数据库日志共同构成 AI 可审计链

## 3.4 业绩库与项目知识复用

这一块在文档里存在感不稳定，但代码里已经是实装能力。

- 前端页面：`frontend/ppa_frontend/src/pages/Assessment/Contracts.tsx`
- 后端接口：`server/routes/contracts.js`
- Dashboard 汇总：`server/services/dashboardService.js`

当前有效认知：

- 系统已经有独立的 CSV 业绩库能力
- 业绩库已进入导航，不是实验脚本
- Dashboard 已经统计业绩数量
- 这条线是后续“标签推荐、相关业绩辅助决策”的基础数据层

## 3.5 Web3D 评估产品线

Web3D 不是一个普通“加了几个字段的表单”，而是并行产品线。

- 前端页面：
  - `frontend/ppa_frontend/src/pages/Web3D/New.tsx`
  - `frontend/ppa_frontend/src/pages/Web3D/History.tsx`
  - `frontend/ppa_frontend/src/pages/Web3D/Detail.tsx`
- 后端：
  - `server/controllers/web3dProjectController.js`
  - `server/services/web3dProjectService.js`
  - `server/controllers/web3dAiController.js`
  - `server/services/web3dAiStep4Service.js`

当前有效认知：

- Web3D 有独立项目 CRUD
- Web3D 有独立风险配置和工作量模板
- Web3D 复用历史、导出等共性骨架
- Web3D Step4 已经接入图片/视觉分析能力

## 3.6 项目机会域

机会域已经从“招标网站清单”演进成一条完整的数据处理流水线。

- 路由入口：`server/routes/opportunity.js`
- 相关服务：
  - `server/services/biddingSiteService.js`
  - `server/services/tenderStagingService.js`
  - `server/services/tenderPushService.js`

已验证能力：

- 招标网站维护
- 脚本批量校验
- 招标暂存库查询
- staging 同步
- 去重预览与执行
- 源文件归档
- 字段解析
- 全网检索
- 小程序推送

当前有效认知：

- 机会域不是单页面辅助功能，而是独立业务域
- `tender_staging` 及其同步逻辑已经构成“机会中台”雏形
- 文档通常只描述其中一部分能力，代码范围更大

## 3.7 小程序与分发链路

目前至少存在两条已验证的推送链路。

- 标准评估项目 -> 内部项目渠道
  - `server/services/pushService.js`
  - 云函数名：`upsertInternalProject`
- 招标暂存项目 -> 小程序招标库
  - `server/services/tenderPushService.js`
  - 云函数名：`upsertTenderBySourceId`

当前有效认知：

- 分发不是手工导出再上传，而是系统能力的一部分
- 评估项目推送和招标推送是两条不同业务链，不应混为一个“推送功能”

## 4. 当前建议视作“已落地”的能力

以下能力可以直接按“已实现系统能力”来理解：

- 标准项目评估主链
- 后端统一计算
- 参数配置中心
- 商务报价相关存储与历史页展示
- PDF / Excel 导出
- AI 风险、模块、工作量、标签能力
- 模型与 Prompt 配置
- AI 日志监控
- Web3D 独立评估线
- CSV 业绩库
- 招标网站与 staging 处理链
- 小程序推送主链

## 5. 当前建议视作“文档存在，但未纳入本次已验证事实范围”的内容

下面这些方向在 `docs/prd` 中存在，但本轮不应直接按“已完整落地”理解：

- 置信度报告推送
- 承建商风险识别
- 行业情报
- 一键工作量评估在 PRD 层面的完整闭环描述
- 小程序会员支付相关全部细节

这些内容里有的已经有部分实现，有的仍更接近路线图。后续若要作为开发依据，建议先做一次“代码对照式确认”。

## 6. 对后续工作的建议

如果后续要继续开发、补文档或做测试，我建议统一采用下面的工作顺序：

1. 先看对应模块代码，确认当前真实行为
2. 再看 `docs/prd/project-overview.md`，确认总口径是否一致
3. 仅在需要业务背景或边界说明时回看分模块 PRD
4. 不再把 `docs/prd/README.md` 当成状态真相，而只把它当导航页

这份地图的核心价值不是“描述所有细节”，而是先把“哪里是真相、哪些能力已经成系统、哪些内容仍需要谨慎对待”固定下来。
