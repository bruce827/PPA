# 小微企业软件项目 Archetype 模板与示例

本文档用于沉淀你在“小微企业软件项目成本评估”方向上的通用分析模板，以及当前确定的三个典型 archetype 骨架，便于后续调研、写报告和扩展 PPA 能力。

## 通用 Archetype 模板（7 段结构）

> 用于描述任何一个“小微企业软件项目”场景的标准模版。

1. **客户画像与业务背景**  
   - 典型企业类型、规模、业务形态  
   - 管理现状与关键矛盾

2. **项目需求表象（客户嘴里说的需求）**  
   - 客户用自己的话是怎么描述“想要一个系统”的  
   - 表面功能诉求与显性动机

3. **深层问题与风险（专业视角）**  
   - 业务/管理/流程层面的本质问题  
   - 软件项目实施中容易出现的关键风险点

4. **市场现有方案 vs 定制开发**  
   - 现成 SaaS / 行业解决方案的大致能力覆盖  
   - 自研/定制开发在该场景下的现实角色与边界

5. **成本评估结论示例（结合 PPA 思路）**  
   - 典型情况下的人天区间与成本区间  
   - 风险放大因子与溢价逻辑

6. **专业建议（做 / 不做 / 怎么做更省钱）**  
   - 是否建议定制，还是优先选 SaaS + 流程/制度改造  
   - 你能提供的评估与咨询价值

7. **对你能力的体现（给自己与投资人看的角度）**  
   - 行业理解与业务洞察  
   - 可解释、可复盘的评估方法论  
   - 可复制的服务形态/产品化潜力

## 案例一：本地培训机构排课系统评估

> 目标：展示“看似要做一个排课系统，其实更需要的是 SaaS 选型 + 管理机制设计”的典型场景。

1. **客户画像与业务背景**  
   - 本地 K12 / 兴趣 / 成人培训机构，门店数与老师数有限  
   - 老板一人管事，缺乏制度与标准流程  
   - 典型乱象：老师可以私下给学生排课，利用老板场地做“私活”

2. **项目需求表象**  
   - 老板口中的需求：“做一个排课系统，规范一下管理”  
   - 期望功能：排课、调课、教室占用、学员上课记录、补课/请假管理等

3. **深层问题与风险（专业视角）**  
   - 本质问题在于管理与机制：  
     - 缺乏统一的排课规则与审批流程  
     - 缺乏对老师行为与收益分配的明确机制  
   - 核心风险：  
     - 做了定制系统但老师可以绕开不用  
     - 规则没想清就开工，需求反复变更

4. **市场现有方案 vs 定制开发**  
   - 市面已有大量培训机构 SaaS，可覆盖大部分排课场景  
   - 一般情况下：SaaS 年费（含实施）总成本显著低于定制开发

5. **成本评估结论示例（结合 PPA）**  
   - 假设坚持做定制排课系统：  
     - 人天区间：使用 PPA 估算（例如 XX–YY 人天）  
     - 金额区间：合理上限“不超过约 7 万”作为锚点  
   - 风险系数：需求变更频繁、管理缺位 → 风险放大

6. **专业建议**  
   - 典型结论：  
     - 不建议直接上定制；优先选型成熟 SaaS + 流程/制度设计  
   - 你提供的价值：  
     - 对比定制 vs SaaS 的 3 年总成本与风险  
     - 帮老板梳理流程与权限/收益机制，确保买的系统能用起来

7. **能力体现**  
   - 看得到“系统需求背后的管理问题”  
   - 能给出合理成本区间与风险结构，而不是凭感觉报价  
   - 形成“劝退型评估”的专业形象：帮客户少花冤枉钱

## 案例二：困境中的小型软件公司——投标前成本与风险评估

> 目标：服务“原有客户断档，只能到各类招标网站梭罗项目”的小型软件公司，在投标前提供独立的成本与风险评估。

1. **客户画像与业务背景**  
   - 10–30 人的小型软件企业 / 外包公司，技术能力尚可，销售与市场薄弱  
   - 原有长期客户减少或断单，现金流紧张，需要快速找项目“养活全公司”  
   - 典型行为：老板/合伙人刷招标网站广撒网，对项目适配度与坑点缺乏系统判断

2. **项目需求表象**  
   - 常见提问：  
     - “这个标我们能不能做？”  
     - “报多少才有机会中标？”  
     - “这个项目值不值得拼一把？”  
   - 实质诉求：投标前的决策支持与成本/报价策略参考

3. **深层问题与风险（专业视角）**  
   - 不清楚自身交付边界：技术栈、团队规模、并行项目能力  
   - 缺乏标准化估算体系：主要依赖拍脑袋与零散经验  
   - 核心风险：  
     - 抢到高风险低利润项目，现金流进一步恶化  
     - 项目需求模糊、甲方强势，导致长期拖期与返工  
     - 无“评估 vs 实际”复盘，估算模型无法进化

4. **市场现有方案 vs 你的评估价值**  
   - 市面常见服务：代写标书、招投标培训等，重点在“如何中标”  
   - 缺位点：很少有人专门站在小软件公司的角度，做**投标前的独立成本与风险评估**  
   - 你的差异：  
     - 不写标书，而是回答：值不值得投 / 成本区间 / 需要多少风险溢价

5. **成本评估结论示例（结合 PPA）**  
   - 对单个招标项目，根据功能规模、接口数量、非功能要求、客户类型、周期等：  
     - 给出人天区间（例如 XX–YY 人天）  
     - 给出成本区间（结合其内部人天单价与风险系数）  
     - 标记风险等级（低/中/高）  
   - 对明显不适合的项目，给出“红灯：不建议投”

6. **专业建议**  
   - 建议建立“投标前必评估”机制：  
     - 分类：可投 / 可投但需溢价 / 坚决不投  
     - 对中标项目做“评估 vs 实际”复盘，迭代估算模型  
   - 你可以提供：  
     - 首批若干招标项目的投标前评估服务  
     - 帮对方基于 PPA 设计内部估算模板与项目分类规则  
     - 定期评估复盘，纠正估算误差

7. **能力体现**  
   - 理解小软件公司的真实生存环境与项目组合风险  
   - 把 ToB 评估经验与 PPA 工具转化为可复制的决策体系  
   - 为将来的“乙方项目组合评估顾问”角色打基础

## 案例三：专业服务小机构——案件/项目管理系统评估

> 目标：服务会计师事务所、律所、咨询工作室等专业服务小机构，评估“是否需要定制案件/项目管理系统”。

1. **客户画像与业务背景**  
   - 小型会计所 / 记账公司、小律所 / 法务工作室、咨询/猎头/HR 服务工作室等  
   - 规模：约 5–30 人，有稳定客户，主要靠人力驱动交付  
   - 管理现状：客户信息、案件/项目进度、合同与应收应付分散在 Excel、微信与个人记忆中

2. **项目需求表象**  
   - 老板通常说：“想做个系统，把客户、合同、进度都管起来。”  
   - 期望能力：案件/项目管理 + 客户与收费管理 + 文档/材料管理

3. **深层问题与风险（专业视角）**  
   - 本质问题：  
     - 流程不统一：每个人办案/做项目方式不同  
     - 权责不清：谁负责录入与维护数据  
     - 数据结构意识薄弱：命名、归档无规范  
   - 核心风险：  
     - 定制系统上线后录入热情快速下降，回到 Excel  
     - 规则尚未设计完毕就开始开发，导致需求反复变更  
     - 期望“一步到位什么都做”，项目复杂度和成本失控

4. **市场现有方案 vs 定制开发**  
   - 市面方案：  
     - 通用 SaaS：CRM、任务管理、知识库、OA 等  
     - 行业垂直产品：律所案件系统、财税记账系统、项目管理 SaaS  
   - 现实判定：  
     - 单一小机构往往更适合“选型 + 配置 + 流程调整”，而不是从 0 定制全套系统

5. **成本评估结论示例（结合 PPA）**  
   - 若坚持定制：  
     - 可根据用户数、案件/项目复杂度、角色数量、集成数量等，用 PPA 估出人天与成本区间  
     - 对需求不清、流程不稳定的场景，设置较高风险系数  
   - 同时给出“定制 vs 现成 SaaS（含订阅 + 适配成本）”在若干年视角下的总成本对比

6. **专业建议**  
   - 建议路径：  
     - 先从 1–2 条关键业务流程做 MVP 级梳理  
     - 尽量通过 SaaS + 流程/制度调整解决 70–80% 问题  
     - 只有在确有差异化需求且具备长期投入能力时才考虑定制  
   - 你可以提供：  
     - 业务流程与数据结构梳理  
     - 定制 vs SaaS 的总成本与风险评估  
     - 对“先做什么、不做什么”给出清晰建议

7. **能力体现**  
   - 把“做个系统”的模糊诉求拆解为流程、数据结构、角色权限与成本风险  
   - 将软件项目评估能力迁移到专业服务行业场景  
   - 通过 PPA 提供可解释、可复盘的评估过程，为未来服务产品化打基础
