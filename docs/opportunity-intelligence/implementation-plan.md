# 项目机会智能评估服务实施路线

## 1. 实施原则

1. 先做服务，再做界面
2. 先做结构化输出，再做复杂模型
3. 先做规则评分，再做学习排序
4. 先打通 PPA 内部调用，再扩展为外部服务

## 2. 分阶段路线

## Phase 0：文档与资产准备

### 目标

完成统一需求、PRD、架构方案和数据口径梳理。

### 交付物

- 需求总结
- PRD
- 技术架构
- 实施路线

### 当前状态

本目录文档即为该阶段产物。

## Phase 1：Headless 智能分析服务 MVP

### 目标

构建一个无 UI 的机会智能评估服务 MVP，完成“输入理解 -> 标准化 -> 检索 -> 评分 -> 输出”闭环。

### 重点任务

1. 定义统一输入输出协议
2. 定义 `opportunity_snapshot`
3. 接入至少一种文档理解 provider
4. 复用现有 PPA 历史项目和 CSV 业绩库
5. 实现规则评分器 v1
6. 输出解释型结果

### 预期交付

- 独立服务代码仓或独立服务目录
- 对内 API
- 一组样例数据与回放日志

## Phase 2：与 PPA 集成

### 目标

让现有 PPA 可直接调用该服务，展示机会推荐结果。

### 重点任务

1. 新增 PPA 后端适配接口
2. 新增结果查询接口
3. 定义 PPA 内部使用的展示结构
4. 初步接入每日招标机会分析流

### 预期交付

- PPA 内部调用链打通
- 可在系统内查看推荐结果

## Phase 3：增强检索与模型辅助重排

### 目标

提升推荐质量和解释质量。

### 重点任务

1. 引入 embedding 检索
2. 引入 reranker
3. 改善标签归一和能力识别
4. 扩充历史项目标签回填

## Phase 4：反馈闭环与学习排序

### 目标

将“是否跟进、是否投标、是否中标、利润表现”等真实反馈引入系统。

### 重点任务

1. 增加反馈标签
2. 沉淀正负样本
3. 评估学习排序可行性
4. 升级评分器

## 3. 第一阶段推荐实现顺序

建议按以下顺序推进：

1. 统一数据结构
2. 抽离独立服务骨架
3. 接入文档理解 provider
4. 构建内部检索
5. 落规则评分器
6. 输出解释结果
7. 接入 PPA

## 4. 第一阶段最小 API 建议

### 4.1 分析接口

`POST /analyze`

输入：

- 公告正文
- 附件信息
- 来源元数据

输出：

- 标准化快照
- 特征结果

### 4.2 推荐接口

`POST /recommend`

输入：

- 机会快照

输出：

- fit_score
- fit_level
- matched_projects
- matched_contracts
- reasons

### 4.3 批量接口

`POST /batch/analyze-and-recommend`

输入：

- 当日新增机会列表

输出：

- 批量推荐结果

## 5. 第一阶段数据准备建议

### 5.1 必做

1. 整理历史项目结构字段映射
2. 回填已有项目标签
3. 为 CSV 业绩库生成可检索摘要

### 5.2 选做

1. 建立项目知识卡片
2. 建立历史项目 embedding
3. 建立合同 embedding

## 6. 第一阶段风险

1. 历史标签覆盖偏少，需要补数据
2. 不同模型输出格式不稳定，需要统一标准化
3. 附件质量差时可能影响结构化质量
4. 如果过早把所有逻辑交给 prompt，会导致系统不可控

## 7. 第一阶段完成标志

满足以下条件即可认为第一阶段完成：

1. 服务能够接收真实招标输入
2. 服务能够输出稳定结构化结果
3. 服务能够结合内部历史数据给出推荐分数
4. 结果可解释
5. PPA 能通过接口调用该服务

## 8. 下一步建议

文档完成后，建议直接进入以下工作流：

1. 根据本目录文档创建独立技术方案 backlog
2. 明确服务目录结构
3. 先搭建 API 与数据模型骨架
4. 再决定是新仓库还是放入当前仓库的 `server/services` 扩展
