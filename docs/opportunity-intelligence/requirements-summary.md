# 项目机会智能评估服务需求总结

## 1. 背景

当前 PPA 已具备较完整的软件项目评估能力，系统中已经沉淀了以下可复用资产：

- 历史项目评估数据
- `projects.assessment_details_json`
- `projects.tags_json`
- 风险、工时、成本、评分因子等结构化评估结果
- 本地 CSV 合同/业绩库
- AI 风险评估、模块分析、工作量评估、项目标签生成接口

同时，项目外部已经存在一个独立爬虫系统，可持续抓取全网当日招标公告。新的目标不是重新做采集，而是利用现有评估资产和外部 AI 能力，对招标机会做自动理解、筛选、评分、排序与推荐。

## 2. 最终目标

构建一个“项目机会智能评估服务”，对每日新增招标信息进行自动分析，输出适合公司承接的机会清单、评分结果和解释说明。

该服务需要：

- 接收原始招标信息与附件
- 利用 AI 完成文档理解和特征抽取
- 结合 PPA 历史项目与业绩库进行匹配
- 使用自定义规则评分输出最终推荐结果
- 以 API 方式提供给当前 PPA 与未来外部系统调用

## 3. 本阶段已达成的关键共识

### 3.1 服务形态

本阶段实现的不是“训练一个自己的大模型”，而是一个独立服务。

推荐名称：

- 项目机会智能评估服务
- 招标机会分析与推荐服务

### 3.2 AI 的职责边界

AI 负责：

- 招标正文/PDF/图片/附件理解
- 字段抽取与结构化
- 项目标签生成与归一
- 风险信号和能力信号提取
- Top N 候选机会的补充解释

AI 不负责：

- 最终推荐主评分
- 最终排序主链
- 最终适配裁决

### 3.3 评分核心

最终推荐必须由代码控制的规则评分器完成，而不是完全交给 prompt。

初始版本的评分逻辑应基于：

- 行业/业务域匹配度
- 技术能力匹配度
- 模块相似度
- 风险适配度
- 成本/规模相似度
- 历史业绩命中度
- 时间与交付压力惩罚

### 3.4 数据表达方式

最终权威结果必须是结构化数据。

需要同时保留两类数据：

- 结构化数据
  - 作为业务真相、审计依据、评分输入和输出
- 向量数据
  - 作为召回、语义检索和相似度匹配的辅助能力

结论：

- 结构化是主存
- 向量化是辅存

## 4. 第一阶段范围

### 4.1 In Scope

1. 接收爬虫已经采集完成的招标机会数据
2. 解析网页正文、PDF、图片、附件中的关键信息
3. 将机会信息统一整理为标准化快照
4. 基于历史项目和业绩库进行检索与相似匹配
5. 生成结构化特征、标签、风险信号和能力信号
6. 计算推荐分数、排序结果与解释说明
7. 对内提供给 PPA 调用，对外预留标准 API

### 4.2 Out of Scope

1. 全网招标采集系统本身
2. 完整多租户权限体系
3. 从零训练自有基础大模型
4. 自动代替人工做最终投标决策
5. 第一阶段复杂监督学习或在线学习排序

## 5. 关键输入资产

### 5.1 外部输入

- 爬虫抓取的招标公告正文
- 招标 PDF、扫描件、图片、附件表格
- 来源站点基础信息

### 5.2 内部输入

- 历史项目 `projects`
- 评估快照 `assessment_details_json`
- 标签 `tags_json`
- CSV 合同/业绩库
- 配置项与角色/风险体系

## 6. 关键输出

服务最终应输出统一结构，至少包括：

- 机会基础摘要
- 项目标签
- 结构化要素
- 相似历史项目
- 相似业绩条目
- 风险信号
- 能力匹配信号
- 推荐分数
- 推荐等级
- 推荐理由
- 不推荐原因

## 7. 第一阶段推荐实现策略

### 7.1 基础形态

第一阶段采用“Headless AI Analysis Service + 规则评分器”方案：

1. AI 负责理解和整理
2. 检索层负责找证据
3. 规则评分器负责最终裁决

### 7.2 为什么不直接做纯 Prompt/RAG 决策

纯 Prompt/RAG 服务可以快速验证思路，但存在以下问题：

- 结果稳定性不足
- prompt 调整后可能漂移
- 难做版本对比
- 难审计和回放
- 难沉淀成长期壁垒

因此，纯 Prompt/RAG 适合作为第一阶段的“智能分析内核”，但不适合作为最终的推荐主评分链路。

## 8. 第一阶段服务定义

建议将能力拆为以下模块：

1. `document-intelligence`
   - 解析与识别多源文档
2. `opportunity-normalizer`
   - 统一招标机会结构
3. `feature-service`
   - 标签、实体、向量、风险/能力特征生成
4. `recommendation-service`
   - 候选召回、规则评分、排序
5. `explanation-service`
   - 输出推荐原因与证据摘要

MVP 可以先部署为一个服务，逻辑上按以上模块分层。

## 9. 成功标准

第一阶段不以“模型训练效果”作为核心衡量，而以“可用性、可解释性、可接入性”作为主标准：

1. 能稳定接收招标数据并输出标准化结构
2. 能返回可复核的评分与排序结果
3. 能解释为什么推荐或不推荐
4. 能被 PPA 后端稳定调用
5. 能作为未来独立服务继续演进
