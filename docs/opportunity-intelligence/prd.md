# 项目机会智能评估服务 PRD

## 1. Executive Summary

项目机会智能评估服务用于接收全网每日新增的招标机会数据，并结合 PPA 中的历史项目评估数据、项目标签和本地业绩库，输出“是否值得跟进、适不适合承接、为什么推荐”的结构化结果。

本服务第一阶段不追求“训练自己的基础模型”，而是以“服务编排 + 文档理解 + 检索匹配 + 规则评分”为主线，快速建立一个可被 PPA 复用、也可对外提供 API 的机会推荐能力。

## 2. Problem Statement

当前存在以下业务痛点：

1. 全网招标机会数量大，人工筛选成本高
2. 是否适合承接高度依赖个人经验，缺少统一判断口径
3. 历史评估经验沉淀在 PPA 中，但尚未反向用于招标机会筛选
4. 招标正文、PDF、图片、附件等多源数据结构不统一，人工理解成本高
5. 现有 AI 能力已能辅助评估，但尚未形成针对招标机会的统一服务

## 3. Product Goal

构建一个“面向招标机会”的智能评估服务，完成以下闭环：

1. 理解输入机会
2. 对齐内部经验资产
3. 形成统一特征表达
4. 输出推荐分数、排序和解释

## 4. Product Positioning

### 4.1 本质定位

本产品是“服务中台能力”，不是单一页面，也不是单一模型。

### 4.2 产物定位

第一阶段产物是：

- 一个可被程序调用的 headless 服务
- 一套标准结构化输出协议
- 一套初始规则评分逻辑

## 5. Users

### 5.1 Primary User

- 售前/项目评估负责人
- 负责初筛每日招标机会，判断是否进入后续评估流程

### 5.2 Secondary User

- PPA 系统本身
- 后续独立招标推荐系统
- 未来外部第三方系统

## 6. Success Criteria

### 6.1 Business Success

1. 每日新增招标机会可自动生成可读分析结果
2. 推荐列表能显著减少人工初筛时间
3. 推荐结果具备解释性，便于人工复核

### 6.2 Product Success

1. 单条机会可在可接受时间内完成解析和评分
2. 输出结果结构稳定，可复算、可追踪
3. 能在 PPA 内部稳定接入

### 6.3 Phase 1 Recommended Metrics

1. 招标机会结构化成功率 ≥ 90%
2. 有效推荐结果生成率 ≥ 95%
3. 人工评审“推荐有参考价值”的比例 ≥ 70%
4. 单条机会分析耗时在合理阈值内可接受

## 7. Scope

### 7.1 MVP Scope

1. 接收外部爬虫产出的招标机会数据
2. 支持网页正文、PDF、图片等输入的 AI 理解
3. 生成统一机会快照
4. 从内部历史项目与业绩库召回候选案例
5. 基于规则输出分数、等级和解释
6. 通过 API 返回结果

### 7.2 Post-MVP

1. 用户反馈闭环
2. 投标结果回流
3. 学习排序与模型重排增强
4. 多租户与权限体系
5. 外部开放服务治理能力

## 8. Functional Requirements

### FR-1 输入接收

1. 服务必须支持接收来自爬虫系统的招标机会主数据
2. 服务必须支持传入网页正文、文件链接、附件元数据、站点来源信息
3. 服务必须支持批量处理每日新增机会

### FR-2 文档理解

1. 服务必须支持调用外部 AI 解析 PDF/图像/正文
2. 服务必须输出统一的结构化抽取结果
3. 服务必须保留原始抽取结果和抽取摘要，供排错与回放使用

### FR-3 标准化快照

1. 服务必须将多源输入整理为统一的 `opportunity_snapshot`
2. 快照至少包含：
   - 基本信息
   - 行业与业务域
   - 采购方/客户类型
   - 技术关键词
   - 预算/规模信号
   - 时间与交付压力
   - 风险线索

### FR-4 内部知识匹配

1. 服务必须能基于内部项目与业绩库做候选召回
2. 至少支持：
   - 历史项目相似度匹配
   - 业绩库命中匹配
3. 匹配结果必须可解释

### FR-5 特征生成

1. 服务必须生成标签、能力信号、风险信号、项目类型信号
2. 服务必须同时支持：
   - 结构化特征
   - 向量化特征
3. 结构化特征是评分主输入，向量化特征是检索辅助输入

### FR-6 评分排序

1. 服务必须基于自定义规则生成最终评分
2. 服务必须输出排序结果
3. 服务必须支持推荐等级划分，例如：
   - 高度推荐
   - 可跟进
   - 谨慎评估
   - 不建议投入

### FR-7 解释输出

1. 服务必须输出推荐理由
2. 服务必须输出风险/不推荐原因
3. 服务必须标明参考了哪些内部历史项目和业绩证据

### FR-8 API 服务化

1. 服务必须提供对内 HTTP API
2. 服务必须预留对外 API 能力
3. 服务必须支持同步查询和批量异步分析

## 9. Non-Functional Requirements

### NFR-1 可解释性

最终评分必须可追溯到结构化字段、规则权重和证据来源。

### NFR-2 可扩展性

模型供应商必须可替换，不能绑定单一厂商。

### NFR-3 可复算性

对同一机会，在相同输入和相同配置下，结果应可复算。

### NFR-4 可治理性

服务必须保留原始输入、标准化结果、评分输出和调用日志。

### NFR-5 安全性

内部项目数据和业绩库不得被无控制地发送给模型，必须经过最小必要裁剪。

## 10. Data Strategy

### 10.1 结构化主数据

最终推荐服务必须以结构化字段作为主数据表达方式。

### 10.2 向量化辅助数据

向量化数据仅用于：

- 相似召回
- 语义检索
- Top N 候选重排

### 10.3 RAG 的定位

RAG 在本产品中仅作为“找证据”的能力，不直接替代最终推荐评分器。

## 11. Phase 1 Key Decisions

1. 不从零训练大模型
2. 先做 Headless Service
3. 先做规则评分，不做学习排序
4. 外部 AI 负责理解，内部服务负责裁决
5. 输出必须是结构化结果

## 12. Risks

1. 历史样本规模有限，短期不适合复杂监督学习
2. 多模型输出存在波动，需要标准化和兜底
3. 附件质量差时，文档理解可能不稳定
4. 纯 Prompt/RAG 容易漂移，必须有规则层收口

## 13. Deliverables

第一阶段交付物包括：

1. 机会智能评估服务 MVP
2. 标准机会快照数据结构
3. 规则评分器 v1
4. 内部知识召回能力
5. 面向 PPA 的集成接口

## 14. Open Questions

1. 第一阶段是否需要落独立数据库，还是先复用现有数据库与新表
2. 对外 API 是否需要鉴权与配额治理
3. 历史项目标签回填是否作为 MVP 前置任务
4. 是否在第一阶段引入向量库，还是先本地 FAISS
