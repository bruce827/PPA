# PRD：项目推送置信度增强 — 阶段一

> **版本**: 1.0
> **日期**: 2026-04-04
> **状态**: 待评审

---

## 1. 背景与目标

### 1.1 背景

PPA 系统已完成 Web 端评估 → 小程序内部渠道推送链路。当前推送包含商务报价、风险评分、工作量、附件等信息，但缺乏对外部市场情报的整合。售前决策者无法在推送时回答两个关键问题：

1. **这个项目值不值得投入售前精力？** — 是否已有中标方？是否是内定项目？
2. **我们的报价在市场中处于什么水平？** — 同类项目别人花了多少钱？

### 1.2 目标

在推送流程中叠加外部市场情报数据，帮助决策者快速判断项目价值与报价合理性，**排除不值得投入的项目**（已完工走账型、内定陪跑型、亏钱型）。

### 1.3 核心用户

- **唯一用户**: 售前/销售负责人（bruce）
- **使用场景**: 推送项目到小程序内部渠道前，在 Web 端查看项目对标数据

### 1.4 设计原则

- **人工筛选优于全量自动化** — 用户仅在项目机会页面对自己认为有价值的项目生成 AI 简报，避免无效 token 消耗
- **推送时查询，非实时计算** — AI 简报预先生成并结构化存储，推送时仅需查询匹配，响应 < 2 秒
- **渐进式建设** — 阶段一只做价格对标，阶段二做签约方风险检测，阶段三做 AI 口碑/潜规则分析

## 2. 功能需求

### FR-1 招标公告 AI 简报生成

#### FR-1.1 项目机会页面新增"生成简报"按钮

- **位置**: 项目机会列表（`/opportunity/bidding-sites` 或 `/opportunity/tender-push`）操作列
- **前置条件**: 该招标项目尚未生成简报
- **触发**: 点击按钮，调用 AI 分析接口
- **状态展示**: 
  - 未生成: 显示"生成简报"按钮
  - 生成中: 显示 loading 状态
  - 已生成: 显示"查看简报"按钮

#### FR-1.2 AI 分析招标公告

调用 AI 模型（豆包/OpenAI）对招标公告全文进行分析，输出结构化简报 JSON：

```json
{
  "tender_id": "原招标数据 ID",
  "projectName": "XX集团绩效考核管理系统建设",
  "projectType": "软件开发",
  "modules": ["指标管理", "绩效评估", "审批流", "数据看板", "权限管理"],
  "budgetAmount": 120,
  "budgetUnit": "万元",
  "clientType": "国企",
  "clientIndustry": "能源",
  "techStack": ["Java", "Vue", "MySQL"],
  "projectScale": "中型",
  "deliveryMonths": 6,
  "tags": ["绩效考核", "数据看板", "审批流"],
  "summary": "某能源集团建设覆盖集团-子公司-部门三级的绩效管理系统，包含指标管理、绩效评估、审批流等模块...",
  "procurementMethod": "公开招标",
  "announcementDate": "2026-03-15",
  "clientName": "XX集团有限公司"
}
```

#### FR-1.3 简报存储

- **表名**: `tender_briefs`
- **结构**:
  | 字段 | 类型 | 说明 |
  |------|------|------|
  | `id` | INTEGER PK | 自增主键 |
  | `tender_id` | INTEGER | 关联 opportunity_tender_push 表 |
  | `brief_json` | TEXT | AI 生成的结构化简报 JSON |
  | `created_at` | DATETIME | 生成时间 |
  | `updated_at` | DATETIME | 更新时间 |

- **索引**: `idx_tender_briefs_tender_id(tender_id)`

#### FR-1.4 简报查看

- 点击"查看简报"按钮，弹出 Modal 展示结构化简报内容
- 展示方式：表格化展示各字段，摘要全文可滚动
- 支持编辑修正（AI 分析可能有误，用户可手动修正字段）

### FR-2 推送时价格对标

#### FR-2.1 推送 Modal 新增价格对标卡片

在现有 PushModal 中，校验通过后、预算输入前，展示价格对标信息：

```
┌─────────────────────────────────────────┐
│  📊 市场招标对标（仅含已分析项目）        │
│                                         │
│  当前报价：89 万                        │
│  同类项目中位价：120 万（3个项目）        │
│  价格区间：85万 - 150万                  │
│                                         │
│  参考项目：                              │
│  1. XX集团绩效考核 - 预算120万  [简报]   │
│  2. YY公司人力数字化 - 中标150万 [简报]  │
│  3. ZZ局员工考核 - 预算85万    [简报]   │
└─────────────────────────────────────────┘
```

#### FR-2.2 对标匹配逻辑

**匹配维度**（按优先级）:
1. **标签交集匹配** — 当前项目标签与招标简报 tags 字段的交集数量
2. **模块关键词匹配** — 当前项目模块名与招标简报 modules 字段的模糊匹配
3. **客户类型匹配** — 同类型客户（国企/政府/民企）优先

**返回规则**:
- 最多返回 5 个同类项目
- 按匹配度降序排列
- 若无匹配项目，展示"暂无同类项目参考"

#### FR-2.3 价格统计计算

- **中位价**: 匹配项目预算/中标金额的中位数
- **价格区间**: P25 - P75 分位
- **对比结论**:
  - 当前报价 < P25 → "低于市场区间，可能具有竞争力或低估成本"
  - P25 ≤ 当前报价 ≤ P75 → "在市场合理区间内"
  - 当前报价 > P75 → "高于市场区间，建议确认是否包含额外范围"

#### FR-2.4 数据来源

| 数据源 | 匹配方式 | 说明 |
|--------|----------|------|
| 业绩库（已有） | 标签单词匹配 | 1000 条结构化数据，直接查询 |
| 招标简报库（新增） | 标签交集 + 模块匹配 | 仅含用户手动生成简报的项目 |

两段结果**合并展示**，标注来源（"内部业绩" / "市场招标"）。

## 3. 技术架构

### 3.1 数据流

```
┌─────────────────────────────────────────────┐
│ 离线阶段（用户手动触发）                      │
│                                             │
│ 项目机会页面 → 点击"生成简报"                 │
│   → 调用 POST /api/tender-briefs/:id         │
│   → AI 分析招标公告全文                      │
│   → 结构化简报存入 tender_briefs 表          │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ 推送阶段（用户点击推送时）                    │
│                                             │
│ 推送 Modal 打开 → 调用 GET                   │
│   /api/projects/:id/market-benchmark         │
│   → 匹配业绩库 + 招标简报库                  │
│   → 返回价格统计 + 参考项目列表              │
└─────────────────────────────────────────────┘
```

### 3.2 新增 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/tender-briefs/:tenderId` | 生成招标简报 |
| GET | `/api/tender-briefs/:tenderId` | 获取招标简报 |
| GET | `/api/tender-briefs` | 获取已生成简报列表 |
| PUT | `/api/tender-briefs/:id` | 编辑修正简报字段 |
| GET | `/api/projects/:id/market-benchmark` | 获取市场价格对标数据 |

### 3.3 前端改动

| 文件 | 改动 |
|------|------|
| `pages/Opportunity/TenderPush/` 或 `BiddingSites/` | 新增"生成简报"/"查看简报"按钮 |
| `components/PushModal.tsx` | 新增价格对标卡片区域 |
| `services/assessment/pushApi.ts` | 新增 marketBenchmark API 函数 |
| `services/opportunity/` | 新增 tenderBriefs API 函数 |

### 3.4 数据库迁移

```sql
CREATE TABLE tender_briefs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    tender_id INTEGER NOT NULL,
    brief_json TEXT NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_tender_briefs_tender_id ON tender_briefs(tender_id);
```

## 4. 非功能需求

### 4.1 性能
- AI 简报生成: 单条分析 < 10 秒（AI 调用耗时）
- 价格对标查询: < 2 秒（纯数据库查询）
- 单用户系统，无需考虑并发

### 4.2 成本
- AI 简报: 用户手动触发，按需消耗 token
- 预估: 每条公告约 1000-3000 tokens，按豆包价格约 ¥0.01-0.03/条

### 4.3 可用性
- AI 分析失败时，允许用户手动填写简报字段
- 招标数据匹配为空时，展示"暂无同类项目参考"，不阻塞推送

## 5. 后续阶段规划

### 阶段二：签约方风险检测
- 用户手动收集签约公司名称
- 接入企查查 API，查询工商/司法风险数据
- 股权穿透分析上下游商务关系
- 推送时展示"签约方风险卡片"

### 阶段三：AI 口碑/潜规则分析
- AI 分析论坛/小红书/知乎等平台上的行业评价
- 识别"行业潜规则"（如国网外包模式、孤岛开发等）
- 结合项目标签，自动关联相关评价
- 推送时展示"行业情报卡片"
