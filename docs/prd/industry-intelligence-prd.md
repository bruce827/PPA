# PRD：AI 行业口碑与潜规则分析（阶段三）

> **版本**: 1.0
> **日期**: 2026-04-04
> **状态**: 待评审

---

## 1. 背景与目标

### 1.1 背景

阶段一解决了"同类项目别人花多少钱"，阶段二解决了"签约方靠不靠谱"。但还有一类信息是任何官方数据和工商 API 都查不到的——**行业潜规则**。

这类信息的特征是：
- 官方数据完全正常（项目真实、手续齐全、额度合理）
- 但实际执行中有很多"坑"
- 这些信息散落在论坛、小红书、知乎、行业交流群的口碑中
- 只有做过同类项目的人才知道

### 1.2 真实案例

| 行业/客户 | 表面信息 | 实际情况（潜规则） | 对项目的影响 |
|-----------|----------|-------------------|-------------|
| 国网项目 | 额度合理、手续正规 | 实际全部外包，验收标准由甲方制定 | 交付压力在外包方，但验收标准是国网的 |
| 港口项目 | 需求清晰、预算充足 | 需要集中开发在一个孤岛上 | 差旅成本暴涨、人员留存率下降、交付周期拉长 |
| 政府年终突击花钱 | 预算充足、急需上线 | 需求不明确、可能只是走流程付款 | 售前投入浪费，项目可能无实际交付要求 |
| 医院信息化项目 | 招标参数完整 | 实际已有指定供应商，只是走形式招标 | 陪跑，浪费时间 |

### 1.3 核心用户

- **唯一用户**: 售前/销售负责人（bruce）

### 1.4 目标

1. 基于项目标签和行业分类，AI 自动搜索并分析论坛/小红书/知乎等平台的口碑评价
2. 识别行业潜规则、交付陷阱、隐性成本
3. 结合已有标签生成功能和业绩库推荐，给出"行业情报卡片"
4. 在推送时展示，辅助最终决策

### 1.5 设计原则

- **标签驱动搜索** — 利用已有的项目标签系统，用标签组合作为搜索关键词
- **AI 聚合，非单条展示** — 不是把搜到的帖子一条条列出来，而是 AI 综合多条信息后给出结论
- **可验证来源** — 每条情报都要标注来源平台和链接，方便用户自行验证
- **渐进式积累** — 每次推送的情报缓存到本地，后续同类项目可直接复用

## 2. 功能需求

### FR-1 行业情报搜索与分析

#### FR-1.1 数据来源

| 平台 | 获取方式 | 数据类型 | 示例搜索词 |
|------|----------|----------|-----------|
| 知乎 | AI 搜索/爬取 | 行业问答、经验分享 | "国网 软件外包 经验" |
| 小红书 | AI 搜索 | 职场吐槽、项目经历 | "驻场开发 孤岛" |
| 脉脉 | AI 搜索 | 员工爆料、公司评价 | "XX公司 项目交付" |
| CSDN/掘金 | AI 搜索 | 技术分享、踩坑记录 | "港口 信息化 踩坑" |
| 政府采购投诉平台 | 爬取 | 投诉/质疑记录 | "XX项目 投诉" |
| 贴吧 | AI 搜索 | 行业讨论、职场吐槽 | "驻场 外包 坑" |

#### FR-1.2 AI 搜索触发方式

**方式一：推送时自动触发**
- 用户点击推送 → 系统根据项目标签 + 行业 + 客户类型自动生成搜索词
- AI 执行搜索 → 分析 → 生成情报卡片
- 耗时: 预计 15-30 秒（多个平台搜索 + AI 分析）

**方式二：手动触发（推荐）**
- 项目详情页新增"行业情报"Tab
- 用户点击"搜索行业情报"按钮 → AI 执行搜索分析
- 结果缓存，后续推送时直接复用

**推荐方式二**，因为：
1. 不阻塞推送流程
2. 用户可以选择是否搜索（不是每个项目都值得搜）
3. 结果可复用，同类项目不用重复搜

### FR-2 行业情报缓存与复用

#### FR-2.1 缓存表

```sql
CREATE TABLE industry_intelligence_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    cache_key TEXT NOT NULL UNIQUE,  -- 标签组合的 hash，如 "绩效考核+国企+能源"
    industry TEXT,
    client_type TEXT,
    tags_json TEXT,
    intelligence_json TEXT NOT NULL,
    sources_json TEXT,              -- 来源链接列表
    overall_risk TEXT,              -- low / medium / high
    summary TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    usage_count INTEGER DEFAULT 1   -- 被多少个项目引用
);

CREATE INDEX idx_intel_cache_key ON industry_intelligence_cache(cache_key);
CREATE INDEX idx_intel_cache_tags ON industry_intelligence_cache(industry, client_type);
```

#### FR-2.2 复用逻辑

- 推送时，根据项目标签组合计算 `cache_key`
- 命中缓存 → 直接展示（标注"X 天前采集"）
- 未命中 → 提示"暂无该行业情报，是否搜索？"

### FR-3 与已有功能的联动

#### FR-3.1 标签系统联动

已有标签生成功能（AI 根据评估数据生成项目标签）的输出，直接作为行业情报搜索的关键词：

```
项目评估完成 → AI 生成标签 → 标签组合 → 行业情报搜索词
```

标签示例：`["绩效考核", "国企", "能源集团", "数据治理", "权限管理"]`
搜索词组合：
- `"国网 绩效考核 软件项目"`
- `"能源集团 信息化 踩坑"`
- `"驻场开发 数据治理"`

#### FR-3.2 业绩库联动

业绩库（已有，1000+ 条）中的项目，如果用户之前做过行业情报分析，可以：
- 推送新项目时，自动关联历史项目的行业情报
- 标签匹配度 > 70% 的项目，共享同一份行业情报

## 3. 技术架构

### 3.1 数据流

```
方式一：手动触发（推荐）
┌─────────────────────────────────────────────┐
│ 项目详情页 → "行业情报"Tab → 点击"搜索"      │
│   → 根据标签组合生成搜索词                   │
│   → AI 搜索多平台（知乎/小红书/脉脉等）       │
│   → AI 综合分析，输出情报 JSON               │
│   → 存入 industry_intelligence_cache         │
│   → 展示行业情报卡片                         │
└─────────────────────────────────────────────┘

方式二：推送时自动触发
┌─────────────────────────────────────────────┐
│ 推送 Modal 打开 → 计算 cache_key             │
│   → 命中缓存 → 直接展示                     │
│   → 未命中 → 提示"暂无情报，是否搜索？"       │
└─────────────────────────────────────────────┘
```

### 3.2 新增 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/industry-intelligence/search` | 触发行业情报搜索 |
| GET | `/api/industry-intelligence/:cacheKey` | 获取已缓存的情报 |
| GET | `/api/projects/:id/industry-intelligence` | 获取某项目的行业情报 |

### 3.3 AI 搜索实现

**方案一：AI 搜索 API**
- 使用支持联网搜索的 AI 模型（如豆包 Pro、GPT-4 with browsing）
- 搜索词由标签组合自动生成
- 优点：实现简单，无需自建爬虫
- 缺点：搜索范围受 AI 联网能力限制

**方案二：自建爬虫 + AI 分析**
- 复用已有爬虫框架（`spider/` 目录）
- 新增知乎/小红书/脉脉的搜索爬取
- AI 仅负责分析，不负责搜索
- 优点：搜索范围广、可控
- 缺点：需要维护多个爬虫

**推荐方案一（AI 搜索 API）**，因为：
1. 单用户场景，搜索频率低
2. 无需维护爬虫
3. AI 模型联网能力正在快速提升

## 4. 非功能需求

### 4.1 性能
- 单次行业情报搜索: < 30 秒（多平台搜索 + AI 分析）
- 缓存命中查询: < 1 秒
- 单用户系统，无需考虑并发

### 4.2 成本
- AI 搜索 API: 每次搜索约 3000-5000 tokens，约 ¥0.05-0.10/次
- 预估: 每月 5-10 次搜索，月度成本 < ¥10

### 4.3 可用性
- 搜索失败时不阻塞推送
- 情报为空时展示"暂无该行业情报"
- 缓存数据超过 30 天时，提示用户"是否重新搜索"
