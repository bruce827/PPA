# PRD：签约方风险检测与商务关系图谱（阶段二）

> **版本**: 1.0
> **日期**: 2026-04-04
> **状态**: 待评审

---

## 1. 背景与目标

### 1.1 背景

阶段一解决了"同类项目别人花多少钱"的问题，但没有回答"跟我们签约的这个公司靠不靠谱"。内部渠道推送的项目，甲方本身通常可靠（国企/大厂），但实际签合同的可能是下级子公司、外包公司、甚至过账公司。签约主体的风险直接决定了回款安全性和交付风险。

### 1.2 核心用户

- **唯一用户**: 售前/销售负责人（bruce）
- **使用场景**: 推送项目前，手动收集签约公司名称，查询其工商/司法/股权风险

### 1.3 目标

1. 支持用户手动录入/收集签约公司名称
2. 通过企查查 API 查询工商信息（实缴资本、参保人数、成立时间等）
3. 通过企查查 API 查询司法风险（失信、诉讼、限高等）
4. 通过股权穿透 API 识别最终实控人和关联风险
5. 在推送时展示"签约方风险卡片"，给出 红/黄/绿 三色判断

### 1.4 设计原则

- **手动录入，非自动爬取** — 签约方信息需要用户自己从客户沟通中获取，系统不做自动化猜测
- **按需查询，非预缓存** — 只有推送前才查一次，不需要定时更新
- **企查查为主，天眼查为辅** — 企查查 API 接口成熟、按次计费灵活；天眼查作为备用数据源

### FR-2 企查查 API 集成

#### FR-2.1 工商基础信息查询

**API**: 企查查"企业基本信息"接口

**返回字段**（映射到风险信号）:

| 字段 | 风险判断 | 阈值 |
|------|----------|------|
| `reg_capital`（注册资本） | 绝对值参考 | — |
| `actual_capital`（实缴资本） | 实缴 < 合同金额 10% → 黄灯 | 需结合报价判断 |
| `established_date`（成立日期） | 成立 < 6 个月 → 红灯 | — |
| `staff_count`（参保人数） | 参保人数 = 0 → 红灯 | — |
| `business_scope`（经营范围） | 不含"软件开发/技术服务" → 黄灯 | 关键词匹配 |
| `tax_level`（税务评级） | D 级 → 红灯 | — |
| `company_status`（经营状态） | 非"存续/在业" → 红灯 | — |

#### FR-2.2 司法风险查询

**API**: 企查查"失信核查" + "司法涉诉" + "限高核查" + "开庭公告" + "裁判文书"

**返回字段**:

| 字段 | 风险判断 | 阈值 |
|------|----------|------|
| `is_dishonest`（是否失信） | 是 → 红灯 | — |
| `dishonest_count`（失信记录数） | > 0 → 红灯 | — |
| `restricted_high_consumption`（是否限高） | 是 → 红灯 | — |
| `lawsuit_count_12m`（近 12 个月诉讼数） | > 5 → 黄灯 | — |
| `defendant_in_contract_dispute`（合同纠纷被告） | 是 → 黄灯 | — |
| `execution_count`（被执行记录） | > 0 且未结案 → 黄灯 | — |
| `no_property_execution_count`（终本案件数） | > 0 → 红灯 | 最危险信号 |

#### FR-2.3 起诉/应诉信息查询

**API**: 企查查"开庭公告" + "裁判文书" + "立案信息"

**返回字段**:

| 字段 | 风险判断 | 阈值 |
|------|----------|------|
| `open_court_cases`（开庭公告列表） | 近两年 > 3 次 → 黄灯 | 展示案由、原告、被告、开庭时间 |
| `court_documents`（裁判文书列表） | 败诉记录 > 2 次 → 黄灯 | 展示案号、案由、判决结果、金额 |
| `case_registration`（立案信息列表） | 近 3 个月新立案 > 2 次 → 黄灯 | 展示案号、案由、立案日期 |
| `is_plaintiff_in_contract_dispute`（作为原告起诉客户） | — | 可能说明上游拖欠款项 |
| `is_defendant_in_labor_dispute`（劳动仲裁被告） | > 3 次 → 黄灯 | 说明内部管理有问题 |
| `recent_judgment_amount`（近一年判决金额合计） | > 实缴资本 50% → 红灯 | 赔付能力存疑 |

#### FR-2.4 股权穿透查询

**API**: 企查查"股权穿透图（四层）" + "疑似实际控制人"

**返回字段**:

| 字段 | 风险判断 | 阈值 |
|------|----------|------|
| `actual_controller`（疑似实际控制人） | — | 展示用 |
| `equity_path`（股权路径） | 层级 > 5 层 → 黄灯 | 层级过深说明是"壳中壳" |
| `parent_company`（母公司/集团公司） | 母公司为甲方 → 绿灯"母公司兜底" | — |
| `parent_company_risk`（母公司风险） | 母公司有失信 → 黄灯 | — |
| `affiliate_risk`（关联企业风险） | 同一控制人下有其他高风险公司 → 黄灯 | — |

## 3. 技术架构

### 3.1 数据流

```
用户录入签约方名称
  ↓
调用 POST /api/contractor-risk/check
  ↓
1. 查本地缓存（contractor_risk_cache）
   → 命中：直接返回
   → 未命中：继续
2. 调用企查查 API（工商 + 司法 + 股权）
3. 聚合风险信号，生成红/黄/绿判断
4. 存入缓存 + project_push_records
5. 返回风险卡片数据
```

### 3.2 新增 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/contractor-risk/check` | 查询签约方风险 |
| GET | `/api/contractor-risk/:id` | 获取已缓存的风险报告 |
| GET | `/api/projects/:id/contractor-risk` | 获取某项目的签约方风险 |

### 3.3 新增配置

- `.env` 新增: `QCC_API_KEY`, `QCC_API_SECRET`, `QCC_API_BASE_URL`
- 企查查 API 按次计费，需在 `openapi.qcc.com` 充值

### 3.4 新增数据库迁移

```sql
-- 签约方风险缓存表
CREATE TABLE contractor_risk_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    company_name TEXT NOT NULL UNIQUE,
    basic_info_json TEXT,       -- 工商基础信息
    judicial_risk_json TEXT,    -- 司法风险信息
    equity_structure_json TEXT, -- 股权穿透信息
    risk_level TEXT,            -- red / yellow / green
    risk_signals_json TEXT,     -- 风险信号列表
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_contractor_cache_name ON contractor_risk_cache(company_name);

-- project_push_records 表新增字段
ALTER TABLE project_push_records ADD COLUMN contractor_company_name TEXT;
ALTER TABLE project_push_records ADD COLUMN contractor_risk_level TEXT;
ALTER TABLE project_push_records ADD COLUMN contractor_risk_signals_json TEXT;
```
