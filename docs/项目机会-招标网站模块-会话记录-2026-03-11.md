## 3. 本次会话中确认的关键规则

### 3.1 URL 规范化规则

- 记录粒度按“平台”而不是具体 SPA hash 页面
- `normalized_url` 主机名统一转小写
- 去掉 `#fragment`
- 去掉默认端口 `:80`、`:443`
- 非默认端口保留
- query 参数全部保留
- 尾部斜杠保留原样
- `www` 不与非 `www` 强行合并
- `http` 和 `https` 不视为同站
- 用户录入 URL 时必须显式带 `http://` 或 `https://`
- 不自动补 `https://`

### 3.2 Seed 入库规则

- 数据源采用“前文清单 + 最终汇总表”
- 汇总入口不入库
- 抓取方案和工具示例链接不入库
- 同一站点冲突时，以最终汇总表为准
- 前文名称补入 `alias_name`
- 增加 `alias_name` 字段
- 第三方综合平台作为正式站点入库
- 若站点仅出现在前文，则按章节推导 `source_level`，缺失字段留空

### 3.3 校验状态模型

- `validation_status` 五态：
  - `never_validated`
  - `validated_ok`
  - `validated_warning`
  - `validated_failed`
  - `heuristic_only`
- `auth_required` 与 `is_bidding_site` 为三态：
  - `true`
  - `false`
  - `null`

### 3.4 网站探测规则

- 仅允许 `http`、`https`
- 后端探测支持 `gzip`、`deflate`、`br`
- 支持非 UTF-8 编码识别和解码
- 至少覆盖 `GBK`、`GB2312`、`GB18030`
- AI 不可用时，仍然返回启发式结果，不让按钮失效

## 7. 本次会话中发现并修复的问题

### 问题：旧数据库启动后未自动建表

用户在接口访问时遇到以下错误：

- `SQLITE_ERROR: no such table: opportunity_bidding_sites`

根因：

- 服务启动时只执行了数据库连接初始化
- 新模块虽然写了建表和 migration，但旧库运行时不会自动补表

修复方式：

- 在 `server/models/biddingSiteModel.js` 中增加 `ensureSchema()`
- 在 `server/index.js` 启动时主动调用 `biddingSiteModel.ensureSchema()`
- 在 `server/seed-data/seed-bidding-sites.js` 中改为先确保表存在，再执行 seed

修复结果：

- 当前数据库已存在 `opportunity_bidding_sites`
- 当前数据库中已成功写入 `92` 条默认数据
- 页面已能正常显示数据
